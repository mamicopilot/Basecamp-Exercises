# 受講ノート — Claude Code Workshop

## このセッションのねらい
自分では書いていない実在のフルスタックアプリ（Vue 3 + FastAPI の在庫管理システム）を題材に、Claude Code を「開発パートナー」として探索・改修・拡張する一連のワークフローを体験する。唯一の正解はなく、計画 → 実装 → 検証のループを回す感覚を身につけることが目的。

---

## キーコンセプト
- **Claude Code = 開発パートナー** — 平易な言葉で意図を伝えれば、コードを読み・計画し・実装・検証まで担う。
- **未知のコードベースの高速オンボーディング** — 構造探索・アーキテクチャ図生成で短時間に全体像をつかむ（ステップ 5）。
- **計画 → 実装 → 検証のループ** — `Shift+Tab` のプランモードで先に計画を承認し、実装後はブラウザで検証して反復する。
- **永続メモリ（CLAUDE.md）** — プロジェクト共有・個人・トピック別ルールを階層で持ち、Claude に継続指示を与える。
- **コンテキストは有限資源** — `/context` で内訳を見て `/compact` で圧縮。MCP もコンテキストを消費する。
- **MCP で能力拡張** — Playwright MCP でブラウザ自動操作など、ローカルの枠を超えた外部ツール連携。
- **GitHub 連携** — 自動コードレビュー・`@claude` タグ・PR ワークフローまでが地続き。
- **高度ワークフロー** — スキル / サブエージェント / フック / プラグイン / エージェントチーム / ワークツリーで反復可能・並行・自動化を実現。

## Claude Code 実践チートシート（よく使う依頼パターン）

| 操作 | 入力 |
|---|---|
| 起動 / モデル選択 | `claude` / `/model` |
| ファイル参照 | プロンプト内で `@CLAUDE.md` |
| メモリに直接書く（無料） | `#` を押す → `# Always document non-obvious logic changes with comments` |
| プランモード切替 | `Shift+Tab` |
| Bash モード | `!` を押す → `! claude mcp add playwright npx @playwright/mcp@latest` |
| コンテキスト確認 / 圧縮 | `/context` / `/compact`（指示付き: `/compact keep the details of the restocking feature`） |
| MCP 確認 | `/mcp` |
| 再起動 | `/exit` → `claude` |
| GitHub App 連携 | `/install-github-app` |
| フック設定 | `/hooks`（または `~/.claude/settings.json`） |
| プラグイン | `/plugin marketplace add ...` → `/plugin install ...` |
| 設定編集 | `/config` |
| トランスクリプト表示 | `Ctrl+O` |
| 中断 | `Ctrl+C` |
| サーバー起動 / ポート解放 | `/start` / `lsof -ti:3000,8001 \| xargs kill` |

**依頼の型:**「動かして」→「理解させて」→「計画させて（プランモード）」→「実装させて」→「ブラウザで検証」→「押し戻して反復」。

## よくある落とし穴 / ベストプラクティス
- **落とし穴:** プランモードに入り忘れ、いきなり実装される。→ **対策:** 大きめの機能は `Shift+Tab` で計画を先に承認。
- **落とし穴:** 要件を一気に詰め込み、出力が不安定。→ **対策:** コアを先に動かし、連携や装飾は反復で追加。
- **落とし穴:** `AskUserQuestion` の確認質問を曖昧に答える。→ **対策:** 具体的に答える（範囲・優先度など）。
- **落とし穴:** 上流リポジトリで GitHub App を承認。→ **対策:** 必ず**自分のフォーク**で承認（ステップ 1 でフォーク）。
- **落とし穴:** MCP 追加後に再起動せず `/mcp` に出ない。→ **対策:** `/exit` → `claude` で再起動。
- **落とし穴:** コンテキストが詰まってから慌てる。→ **対策:** 早めに `/context`、進行中タスクは `/compact keep ...` で保持。
- **ベストプラクティス:** エラーは全文コピペ、画面崩れはスクリーンショットを貼る（Ctrl+V）。
- **ベストプラクティス:** 生成されたアーキテクチャ図や修正は鵜呑みにせず、実ファイル・ブラウザ・Console で答え合わせ。
- **ベストプラクティス:** 行き詰まったら `think harder` / `ultrathink` で拡張思考をトリガー。
- **ベストプラクティス:** 権限プロンプトは内容を読んでから Enter で承認（拒否は Escape）。

## 持ち帰りポイント
1. **意図を平易に伝えれば Claude が計画から実装・検証まで担う** — 完璧な仕様書は不要。会話で詰める。
2. **計画 → 実装 → 検証 → 反復のループ**が中核。プランモードで先に承認し、結果は必ず自分の目で確認する。
3. **コンテキストと権限を意識的に管理する** — `/context`・`/compact`、権限プロンプトの確認が安定運用の鍵。
4. **MCP・スキル・サブエージェント・GitHub 連携で能力を拡張**でき、反復可能・並行・自動化のワークフローが組める。
5. **押し戻し（フィードバック）が品質を生む** — 思い通りでなければ遠慮なく修正依頼。間違いはなく、体験して感覚をつかむことが目的。
