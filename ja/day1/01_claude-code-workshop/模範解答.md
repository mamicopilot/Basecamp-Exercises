# 模範解答 — Claude Code Workshop

> このセクションは「Vue 3 + FastAPI のファクトリー在庫管理システムを Claude Code で探索・改修・拡張する」**オープンエンド課題**です。自分では書いていないコードベースで Claude Code を開発パートナーとして体験することが目的であり、**唯一の正解（成果物）は存在しません**。
>
> したがって本書は「正解コード」ではなく、**ベストの取り組み方（模範的ウォークスルー）**を示します。12 の中核ステップそれぞれについて、(a) Claude Code に何を依頼すべきか（具体的なプロンプト例）、(b) 良い成果の判断基準、(c) つまずきやすい点と回避策、を提示します。最後にエキスパートチャレンジの模範的アプローチをまとめます。

> **注記（コード未実行）:** 本書のプロンプト・コマンド・出力例は環境依存であり、ここでは実行していません。実際の挙動・ポート・ファイル構成は受講者の環境やリポジトリの版によって異なる場合があります。固有名詞・コマンド・URL・技術名（Vue 3, FastAPI, Playwright, MCP, uv 等）は原文どおり保持しています。

---

## 全体の心構え（ウォークスルーの前提）

- **平易な言葉で意図を伝える。** Claude はコードベースを読み、計画し、実装します。完璧な仕様を書く必要はありません。
- **計画 → 実装 → 検証のループを回す。** 一発で完璧を狙わず、ブラウザで確認し、押し戻し（フィードバック）で反復します。
- **常に「何が起きているか」を把握する。** 権限プロンプトは内容を読んでから承認します。Bash の実行・ファイル変更・ツールインストールは都度確認されます。
- **詰まったら情報を渡す。** エラー全文をコピペ、または画面のスクリーンショットを貼り付け（Ctrl+V / Cmd+V）。

## ステップ 1: リポジトリをフォーク & クローンする（10 pts）

**(a) 何を依頼／実行すべきか**

このステップは基本的に手作業（GitHub UI と git）で行います。Claude Code 起動前なので、まずは自分で実施します。
- 推奨は **オプション 1（Fork + Git）**。後続ステップ（GitHub App 連携・PR）まで体験するために必須です。
  ```
  git clone git@github.com:YOUR_USERNAME/inventory-management.git
  cd inventory-management && git checkout -b new_features
  ```
- ZIP ダウンロード（オプション 2）でも演習自体は進められますが、ステップ 10・11 の GitHub 連携が使えません。

**(b) 良い成果の判断基準**
- フォークが**自分の GitHub アカウント**配下に存在する。
- ローカルで `new_features` ブランチに切り替わっている（`git status` / `git branch` で確認）。

**(c) つまずきやすい点と回避策**
- **SSH 鍵未設定で clone 失敗** → HTTPS URL に切り替えるか、GitHub の鍵設定を済ませる。
- **上流（beck-source）をクローンしてしまう** → 必ず**自分のフォーク**をクローンする。ステップ 10 の GitHub App は自分のフォークで承認する必要があるため、ここを誤ると後で詰まります。

## ステップ 3: 在庫管理アプリをローカルで起動する（15 pts）

**(a) 何を依頼すべきか（プロンプト例）**

原文プロンプトをそのまま貼り付けるのが模範的です。
```
Install dependencies and start the development servers and open up the
frontend and backend in my respective browser windows.
```
日本語で意図を補足したい場合の例:
> 依存関係をインストールして開発サーバー（フロントエンド/バックエンド）を起動し、それぞれをブラウザで開いて。起動できたら使い方ページの一覧も教えて。

**(b) 良い成果の判断基準**
- フロントエンド（Vue 3）の `npm install`、バックエンド（FastAPI）の `uv sync` が完了。
- バックエンドがポート `8001`、フロントエンドがポート `3000` で起動。
- ブラウザで `http://localhost:3000` が開き、Dashboard / Inventory / Orders / Spending / Demand / Backlog の各ページが表示される。

**(c) つまずきやすい点と回避策**
- **ポート競合（3000 / 8001 が使用中）** → `lsof -ti:3000,8001 | xargs kill` で解放、または `/start` を再実行。
- **`uv` 未インストール** → エラー全文を Claude に渡し、インストール手順を任せる。
- このアプリには**意図的なバグが仕込まれている**ので、表示が一部おかしくても正常です（特に Reports ページはエキスパートチャレンジ対象）。
- 数分かけて各ページをクリックし、**何が正常で何が変か**を自分の目で把握しておくと、後のステップが楽になります。

## ステップ 4: CLAUDE.md ファイルを編集する（15 pts）

**(a) 何を依頼すべきか（プロンプト例）**

まず `@` メンションでファイル参照を体験します。
```
Print out exactly what is in @CLAUDE.md
```
そのうえで、永続指示を 1 行追加します。**推奨は方法 1（`#` メモリモード、トークン消費ゼロ）**。
```
# Always document non-obvious logic changes with comments
```
Claude に編集を任せたい場合（方法 3、整理・言い回しを任せられる）:
```
Edit my CLAUDE.md file to add "Always document non-obvious logic changes
with comments"
```

**(b) 良い成果の判断基準**
- `@CLAUDE.md` の内容が出力され、`@` 参照の仕組みを理解できた。
- `CLAUDE.md` に新しい指示行が追加され、保存されている。
- 3 つのメモリ階層（`./CLAUDE.md` プロジェクト共有 / `./.claude/CLAUDE.md` 個人全体 / `./CLAUDE.local.md` 個人ローカル）の違いを理解できた。

**(c) つまずきやすい点と回避策**
- **方法の使い分けを誤る** → 単純な追記は無料の方法 1・2 が効率的。文章整理を任せたいときだけ方法 3。
- **個人設定をプロジェクト共有ファイルに書いてしまう** → 共有したくない個人ルールは `./.claude/CLAUDE.md` または `./CLAUDE.local.md` へ。`/memory` は個人用 `./.claude/CLAUDE.md` に書き込む。

## ステップ 6: 機能を構築する（30 pts）— ワークショップの核心

**共通の進め方:** `Shift+Tab` で**プランモード**に入り、Claude にコードを書く前に計画を提示させ、承認してから実装させます。`AskUserQuestion` ツールで確認質問が来たら都度回答します。

### オプション A: 予算ベースの補充ツール

**(a) プロンプト例**
```
Build a new "Restocking" tab in the app. It should have:
1. A budget slider that lets the user set their available budget
2. Based on the budget, recommend items from the demand forecast to restock
3. A "Place Order" button that submits the restocking order
4. The submitted order should appear in the existing Orders tab with a new
"Submitted Orders" section showing delivery lead time
```

### オプション B: SaaS スタイルの UI リデザイン（スキル活用）

**(a) プロンプト例（パート 1: スキル作成）**
```
I want to build a skill that redesigns a Vue 3 application's UI into a modern
SaaS-style interface with a vertical navigation sidebar on the left instead of
a top nav bar, consistent spacing, and a polished professional look.
```
**（パート 2: スキル適用）**
```
Use the frontend-design skill to redesign this inventory management app into a
modern SaaS-style interface with:
1. Vertical navigation bar on the left side instead of the top
2. Clean, modern card layouts
3. Professional SaaS aesthetic
```
さらに反復:
```
Add a collapsible sidebar with icons-only mode for smaller screens.
```

**(b) 良い成果の判断基準（両オプション共通）**
- 実装**前に**計画が提示され、自分が承認している（プランモードを正しく使えた）。
- ブラウザで実際に動く（A: スライダーで予算を変えると推奨が変わり、注文が Orders タブに反映 / B: 左サイドナビ・カードレイアウト・折りたたみが機能）。
- 1 回で完璧でなくても、**フィードバックで改善できている**。

**(c) つまずきやすい点と回避策**
- **プランモードに入り忘れていきなり実装される** → `Shift+Tab` の状態を確認。計画なしで進むと手戻りが増える。
- **AskUserQuestion を無視/放置** → 質問には具体的に答える（例: スライダー範囲は「データから自動決定」）。曖昧だと実装も曖昧になる。
- **要件を一気に詰め込みすぎる** → まずコア（A ならスライダー＋推奨）を動かし、注文連携などは反復で足すと安定。
- **画面の崩れは口頭説明より画像で** → スクリーンショットを貼って（Ctrl+V）「ここをこう」と指示すると速い。

## ステップ 12: 高度なワークフロー（25 pts）

時間に応じて以下から選んで体験します。**1〜2 個を深く試す**のが、すべて表面的に触るより学びが大きく、模範的です。

- **スキル（再利用可能な指示セット）**
  ```
  I want to build a skill that analyzes Vue component structure and suggests
  optimizations for performance and code reuse
  ```
- **サブエージェント（専門エージェント）**
  ```
  Create a new Debugger subagent that specializes in investigating runtime errors,
  reading stack traces, and suggesting fixes. It should have access to Read, Grep,
  Glob, and Bash tools.
  ```
  ```
  Use the debugger agent to investigate any console errors on the Dashboard page.
  ```
- **フック（自動イベント応答）** — `npm install --save-dev prettier` の後、`/hooks` または `~/.claude/settings.json` で `PostToolUse`（`Edit|Write`）に `prettier --write` を設定。乱れた整形のコードを書かせて自動整形を確認。
- **プラグイン（コミュニティ製ワークフロー）** — `/plugin marketplace add ...` → `/plugin install epcc-workflow` → `/epcc-code ...` で利用。
- **エージェントチーム（協調マルチエージェント、実験的）** — `/config` で `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS: "1"` を設定・再起動し、Security Auditor / Performance Analyst / UX Reviewer の 3 名で並行監査させる。`Shift+Down` で各エージェントを切り替え。
- **ワークツリー（独立並行ブランチ）**
  ```
  Use a worktree to prototype a dark mode toggle for the app without affecting my
  current branch. Create the feature, test it, and show me the result.
  ```

**良い成果の判断基準:** 選んだ機能が動き、「いつこの機能を使うべきか（使いどころ）」を自分の言葉で説明できる。

**つまずきやすい点と回避策**
- **全部を浅く触って消化不良** → 興味のある 1〜2 個に絞って深掘りする。
- **実験的機能（エージェントチーム）はフラグ＋再起動が必要** → 設定漏れに注意。
- **フックのコマンドが OS 依存** → 例の bash 構文は環境に合わせて調整する。

## ★ エキスパートチャレンジ: バグバウンティ — Reports ページを修正する（25 pts）

**模範的アプローチ**

### ステップ 1: まず人間が観察する（約 2 分）
Claude にいきなり丸投げせず、**自分でバグを発見**してから依頼すると精度が上がります。
- 右上の**言語切り替え**で言語を変え、Reports ページが翻訳されるか確認（→ 翻訳されない）。
- **時間帯フィルター（例: "January"）/ 倉庫フィルター（例: "San Francisco"）**を設定し、Reports が反応するか確認（→ グローバルフィルターを無視）。
- **DevTools（F12 / Cmd+Option+I）の Console** を開き、Reports 周辺をクリック（→ `console.log` のスパム）。
- 視覚確認だけで**少なくとも 3 カテゴリ**のバグが見つかるはず。

### ステップ 2: Claude Code に修正させる（約 3〜5 分）
発見した症状を具体的に列挙し、「これ以外にもある」「正常なページを参照に」と伝えるのが模範的です。
```
The Reports page (/reports) has multiple bugs compared to the rest of the app.
I found that:
1. It doesn't translate to Japanese when I switch languages
2. It ignores the global filter bar
3. It spams the browser console with debug logs

Investigate the Reports page code, identify ALL the issues (there are more than
these 3), and fix them. Look at how other pages like Dashboard and Orders are
implemented for reference.
```

**良い成果の判断基準**
- Claude が `Reports.vue` を読み、`Dashboard.vue` / `Orders.vue` など**正常なページと比較**している。
- 提示した 3 件を含め **8 件以上**のバグを特定・修正。
- 修正後、`http://localhost:3000/reports` を更新して: 言語切り替えが効く / フィルターでデータが更新される / `console.log` スパムが消える / コードが **Composition API パターン**に沿う。
- `App.vue` の「Reports」ナビタブの翻訳も修正されている。

**つまずきやすい点と回避策**
- **症状を Claude に渡さず丸投げ** → 自分の観察結果を具体的に伝えると検出率が上がる。
- **3 件で満足してしまう** → 「ALL the issues（8 件以上）」と明示し、参照すべき正常ページを示す。
- **修正後の検証を Claude 任せにする** → 自分でもブラウザと Console を再確認する。

### ストレッチゴール
Backlog ページ（`/backlog`）も同様の i18n 問題（全文字列が `t()` を使わず英語ハードコード）。Reports と同じ要領で、翻訳関数 `t()` を使うようリファクタリングを依頼する。
