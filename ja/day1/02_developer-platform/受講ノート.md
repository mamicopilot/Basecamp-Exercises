# 受講ノート — Developer Platform

題材: TechFlow（1日500件以上のチケットを処理する B2B SaaS）向けの **サポートチケット処理エージェント** を、フレームワークなしで Claude API（Messages API）を直接呼び出して構築する。モデルは `claude-sonnet-4-6`。

---

## このセッションのねらい

- 低レベルの **Messages API** を使い、リクエスト／レスポンスのサイクルを自分で制御してエージェントを組み立てる。
- 3つのツール（`get_ticket` / `search_kb` / `resolve_ticket`）を1つのループで自律実行させる。
- 構造化出力・アダプティブ思考・思考量制御・ストリーミングを **1つのエージェントに合成** する。

> Messages API は最下層。これが分かると Agent SDK / Claude Code / claude.ai など上位レイヤの仕組みが見通せる。

---

## キーコンセプト

| 概念 | 一言で | この演習での現れ方 |
|---|---|---|
| エージェンティックループ (agentic loop) | Claude がツールを呼び→結果を返し→また呼ぶ、を繰り返す | `while response.stop_reason == "tool_use"` |
| ツール使用 (tool use) | Claude に外部関数を呼ばせる仕組み | `tools` スキーマ＋`tool_use`/`tool_result` |
| 構造化出力 (structured output) | 出力を JSON Schema に拘束 | `output_config={"format": RESOLUTION_SCHEMA}` |
| アダプティブ思考 (adaptive thinking) | 複雑さに応じて思考量を自動調整 | `thinking={"type": "adaptive"}`（毎回） |
| 思考量 (effort) | 推論の深さを high/low などで指定 | `output_config={"effort": effort}` |
| ストリーミング (streaming) | トークンを逐次受信 | `client.messages.stream()` |

**2フェーズの黄金パターン**（パート3〜5 共通）:
1. **Phase 1**: ツールループ（`format` は付けない）。
2. **Phase 2**: 最終1回だけ `output_config.format` ＋ `tool_choice={"type": "none"}` で構造化出力。

## API チートシート（最小コード断片）

### ツールスキーマ
```python
{
  "name": "search_kb",
  "description": "...いつ呼ぶべきかを具体的に書く...",
  "input_schema": {
    "type": "object",
    "properties": {"query": {"type": "string", "description": "..."}},
    "required": ["query"]
  }
}
# status は enum 固定: ["resolved", "escalated", "pending_customer"]
```

### エージェンティックループ
```python
resp = client.messages.create(model=MODEL, max_tokens=32000, system=SYSTEM_PROMPT,
                              tools=tools, thinking={"type": "adaptive"}, messages=messages)
while resp.stop_reason == "tool_use":
    messages.append({"role": "assistant", "content": resp.content})  # thinking 含め丸ごと
    results = [{"type": "tool_result", "tool_use_id": b.id,
                "content": execute_tool(b.name, b.input)}
               for b in resp.content if b.type == "tool_use"]
    messages.append({"role": "user", "content": results})
    resp = client.messages.create(..., messages=messages)
```

### 構造化出力（最終コール）
```python
final = client.messages.create(
    ..., messages=messages,
    output_config={"format": RESOLUTION_SCHEMA},
    tool_choice={"type": "none"},
)
result = get_structured_result(final)  # = json.loads(最後の text ブロック)
```

### 思考量制御
```python
output_config={"effort": effort}                      # ループ中
output_config={"effort": effort, "format": SCHEMA}    # 最終コール
```

### ストリーミング
```python
with client.messages.stream(..., output_config={"effort": effort}) as stream:
    for event in stream:
        if event.type == "content_block_delta":
            d = event.delta
            if d.type == "thinking_delta":   print(d.thinking, end="")
            elif d.type == "text_delta":     print(d.text, end="")
            elif d.type == "input_json_delta": print(d.partial_json, end="")
resp = stream.get_final_message()   # ループ継続判定 & 最終 JSON はここから
```

## よくある落とし穴 / ベストプラクティス

- **thinking ブロックを落とさない。** `response.content` を加工せずそのまま `assistant` ターンとして返す。tool_use だけ抜き出すと 400 エラー。
- **`format` はループ中に付けない。** 全 text 出力を縛るため、ツール使用と両立しない。**最後の1回だけ。**
- **最終コールに `tool_choice={"type": "none"}`。** 付け忘れると Claude が再びツールを呼び、構造化テキストを返さない。
- **JSON は「最後の」text ブロック。** adaptive 思考で content は `[thinking, text]` になり得る。`text_blocks[-1]` をパース。
- **`effort` は `output_config` 側。** `thinking={...}` 側ではない。最終コールでは `effort` と `format` を両方 `output_config` に入れる。
- **delta の属性名。** `thinking_delta`→`.thinking` / `text_delta`→`.text` / `input_json_delta`→`.partial_json`。
- **tool_result の取り違え。** `tool_use_id` は `block.id`、`content` はツール戻り値（文字列）、`role` は `user`。複数 tool_use があれば全件返す。
- **`get_final_message()` は `with` ブロックを抜けた後**（ストリーム消費後）に呼ぶ。
- **`max_tokens=32000` と長め `timeout`。** 思考＋ツール＋応答を収め、高 effort のレイテンシにも耐える（`client` は `timeout=900.0`）。

## 持ち帰りポイント

1. **ループは Claude が運転、コードはオーケストレーション。** `while stop_reason == "tool_use"` の枠だけ用意すれば、ツールの呼び出し順は Claude が決める。
2. **thinking は丸ごと返す。** アダプティブ思考の content は加工せずそのまま会話履歴へ。これがエラー回避の最重要ルール。
3. **構造化出力は「最後に1回」。** ツールループ中は `format` を付けず、最終コールでだけ `format` ＋ `tool_choice={"type": "none"}`。
4. **effort で深さを使い分ける。** 曖昧なチケット（例: TKT-1046）は `high`、定型は `low`。500件/日を効率と品質で振り分ける運用設計の起点。
5. **4機能は合成できる。** エージェンティックループ＋構造化出力＋アダプティブ思考＋ストリーミングは、1つの `run_agent_streaming` に同居する。
