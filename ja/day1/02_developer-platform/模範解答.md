## 穴埋め1: ツールスキーマ定義（`tools`）

### (a) 完成コード

```python
tools = [

print(f"Defined {len(tools)} tool schemas: {[t['name'] for t in tools]}")
```

### (b) なぜそうするか（要点解説）

- **`name` はバックエンド関数名と一致させる。** `execute_tool(name, input_data)` は `TOOL_FUNCTIONS`（`{"get_ticket": ..., "search_kb": ..., "resolve_ticket": ...}`）で名前引きするため、スキーマの `name` が関数名と一致していないとツール実行に失敗します。
- **`description` がツール選択の決め手。** Claude は `description` を読んで「いつ・どのツールを呼ぶか」を判断します。曖昧な説明はツール誤選択につながるため、`search_kb` には「解決前に必ず KB を検索する」「最大3件返る」など使いどころを明示しています。
- **`input_schema` は JSON Schema。** `properties`／`required` を正しく宣言することで、Claude が渡す引数（`ticket_id`、`query`、`resolution`、`status`）が関数シグネチャ（`get_ticket(ticket_id)`、`search_kb(query)`、`resolve_ticket(ticket_id, resolution, status)`）と一致します。
- **`status` は enum で値域を固定。** `["resolved", "escalated", "pending_customer"]` に制約することで、下流の escalation router やダッシュボードが期待する3状態以外が入り込みません。バックエンドの `resolve_ticket` も `ticket["status"] = status` で受け取るため、値の表記揺れを防げます。

### (c) つまずきやすい点

- `resolve_ticket` の `required` に `status` を含めること（バックエンドのデフォルトは `"resolved"` だが、スキーマ上は必須にして Claude に明示的判断をさせる）。
- enum の綴りはバックエンドが期待する文字列と完全一致が必要（`escalated` を `escalate` と書かない、`pending_customer` のアンダースコアを忘れない）。
- `description` を空のままにしない。空文字だと Claude が `search_kb` をいつ呼ぶべきか判断できず、ループが意図通り回りません。

## 穴埋め2: エージェンティックループ（`run_agent`）

### (a) 完成コード

```python
def run_agent(user_message: str):
    """Run the support ticket agent."""
    messages = [{"role": "user", "content": user_message}]

    response = client.messages.create(
        model=MODEL,
        max_tokens=32000,
        system=SYSTEM_PROMPT,
        tools=tools,
        thinking={"type": "adaptive"},
        messages=messages,
    )

    # Agentic loop: keep going while Claude wants to use tools
    while response.stop_reason == "tool_use":
        # 1) Append the assistant turn as-is — INCLUDING thinking blocks
        messages.append({"role": "assistant", "content": response.content})

        # 2) Execute every tool_use block and collect the results
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                result = execute_tool(block.name, block.input)
                print(f"  🔧 {block.name}({json.dumps(block.input)}) -> {result[:80]}")
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result,
                })

        # 3) Send the tool results back as the next user turn
        messages.append({"role": "user", "content": tool_results})

        # 4) Call the API again
        response = client.messages.create(
            model=MODEL,
            max_tokens=32000,
            system=SYSTEM_PROMPT,
            tools=tools,
            thinking={"type": "adaptive"},
            messages=messages,
        )

    return response
```

### (b) なぜそうするか（要点解説）

- **エージェンティックループの本質。** `stop_reason == "tool_use"` の間だけループします。Claude が「もうツールは要らない」と判断すると `stop_reason` が `end_turn` などに変わり、最終応答（text ブロック）が返ってループを抜けます。**順序は Claude が決め、コードはオーケストレーションするだけ** です（get_ticket → search_kb → resolve_ticket という流れを Claude 自身が選ぶ）。
- **アシスタントの content を「そのまま」返送する。** `messages.append({"role": "assistant", "content": response.content})` で、`thinking` ブロックも `tool_use` ブロックも丸ごと返します。アダプティブ思考では応答に thinking ブロックが含まれ、これを欠落させると後続呼び出しで整合性エラーになります。
- **`tool_result` の組み立て。** `tool_use_id` に `block.id` を、`content` にツールの戻り値（文字列）を入れ、`role: "user"` のメッセージとして返します。複数の `tool_use` があれば、その全てに対応する `tool_result` を1つの user メッセージにまとめます。
- **`thinking={"type": "adaptive"}` を毎回付ける。** Claude Sonnet 4.6 はタスクの複雑さに応じて思考量を自動調整します。各 `create()` 呼び出しに付与します。
- **`max_tokens=32000`。** 思考トレース＋ツール引数＋最終応答を十分収める大きめの値。ノートブックの `client` は `timeout=900.0` で初期化済み（非ストリーミングで大きな `max_tokens` を使うため）。

### (c) つまずきやすい点

- **thinking ブロックを落とさない。** 「tool_use ブロックだけ抜き出して返す」とすると、直前の thinking ブロックが欠けて 400 エラーになります。`response.content` を加工せずそのまま渡すのが安全。
- **tool_result の対応漏れ。** 1ターンに複数 `tool_use` がある場合、全ての `tool_use_id` に対する `tool_result` を返さないとエラーになります。
- **ループ末尾で API を再呼び出しすること。** ツール結果を append したら必ず `create()` を呼び直す。これを忘れると古い `response` のまま無限ループ／早期 return になります。
- **role の入れ違い。** アシスタントターン（`response.content`）は `role: "assistant"`、ツール結果は `role: "user"` です。

## 穴埋め3: 構造化出力（`run_agent_structured`）

### (a) 完成コード

`RESOLUTION_SCHEMA` と `get_structured_result()` は原典で定義済みです（再掲）。

```python
RESOLUTION_SCHEMA = {
    "type": "json_schema",
    "schema": {
        "type": "object",
        "properties": {
            "diagnosis": {"type": "string", "description": "Root cause analysis of the issue"},
            "solution_steps": {"type": "array", "items": {"type": "string"}, "description": "Ordered steps to resolve"},
            "confidence": {"type": "string", "enum": ["high", "medium", "low"]},
            "escalation_needed": {"type": "boolean"},
            "category": {"type": "string", "enum": ["billing", "technical", "account", "feature_request"]}

        "required": ["diagnosis", "solution_steps", "confidence", "escalation_needed", "category"],
        "additionalProperties": False
    }
}


def get_structured_result(response) -> dict:
    """Extract the structured JSON from the last text block in the response."""
    text_blocks = [b for b in response.content if b.type == "text" and b.text.strip()]
    if text_blocks:
        return json.loads(text_blocks[-1].text)
    return None
```

### (b) なぜそうするか（要点解説）

- **`output_config.format` は全ての text 出力を制約する。** そのためツールループ中には付けません。ループ中に付けると、Claude の途中テキストまで JSON 形式に縛られ、ツール使用と両立しません。**「ループはフォーマット無し」「最後の1回だけフォーマット有り」** が鉄則です。
- **最終コールの2点セット。** `output_config={"format": RESOLUTION_SCHEMA}` で出力を json_schema に拘束し、`tool_choice={"type": "none"}` でこれ以上ツールを呼ばせない（=必ずテキストで答えさせる）ようにします。さらに「Provide your structured resolution as JSON.」という user メッセージで明示的に最終出力を促します。
- **直前のアシスタントターンも append する。** Phase 1 ループを抜けた時点の `response`（最終アシスタントターン）を会話履歴に積んでから、user の依頼メッセージを足します。これで会話が `assistant → user → (final) assistant` と正しく連結されます。
- **パースは最後の text ブロックから。** アダプティブ思考では content が `[thinking, text]` になり得ます。JSON は **最後の text ブロック** にあるため、`get_structured_result()` は `text_blocks[-1]` を `json.loads` します。
- **スキーマの `additionalProperties: False` と `required`。** 余分なキーを禁止し、5フィールド（`diagnosis`, `solution_steps`, `confidence`, `escalation_needed`, `category`）を必須にすることで、下流システムが安心して機械処理できます。

### (c) つまずきやすい点

- ループ中に `format` を付けてしまう（→ ツールが動かない／挙動が崩れる）。
- 最終コールで `tool_choice={"type": "none"}` を忘れる（→ Claude がまたツールを呼び、構造化テキストを返さない）。
- `get_structured_result` が **最後の** text ブロックを見ている点を理解せず、最初の text ブロックを読もうとする。
- enum 値（`confidence`, `category`）の綴りは原文どおり。`feature_request` のアンダースコアに注意。

### (a) 完成コード

```python
def run_agent_thinking(user_message: str, effort: str = "high") -> dict:
    """Run agent with effort-controlled adaptive thinking."""
    messages = [{"role": "user", "content": user_message}]

    # --- Phase 1: tool loop with effort control (NO format yet) ---
    response = client.messages.create(
        model=MODEL,
        max_tokens=32000,
        system=SYSTEM_PROMPT,
        tools=tools,
        thinking={"type": "adaptive"},
        output_config={"effort": effort},
        messages=messages,
    )

    while response.stop_reason == "tool_use":
        # Show the agent's reasoning between tool calls
        for block in response.content:
            if block.type == "thinking" and block.thinking.strip():
                print(f"  💭 {block.thinking.strip()[:300]}")
            if block.type == "tool_use":
                print(f"  🔧 {block.name}({json.dumps(block.input)})")

        messages.append({"role": "assistant", "content": response.content})

        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                result = execute_tool(block.name, block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result,
                })

        messages.append({"role": "user", "content": tool_results})

        response = client.messages.create(
            model=MODEL,
            max_tokens=32000,
            system=SYSTEM_PROMPT,
            tools=tools,
            thinking={"type": "adaptive"},
            output_config={"effort": effort},
            messages=messages,
        )

    # --- Phase 2: final structured call at the same effort level ---
    messages.append({"role": "assistant", "content": response.content})
    messages.append({
        "role": "user",
        "content": "Provide your structured resolution as JSON.",
    })

    final = client.messages.create(
        model=MODEL,
        max_tokens=32000,
        system=SYSTEM_PROMPT,
        tools=tools,
        thinking={"type": "adaptive"},
        output_config={"effort": effort, "format": RESOLUTION_SCHEMA},
        tool_choice={"type": "none"},
        messages=messages,
    )

    return get_structured_result(final)
```

### (b) なぜそうするか（要点解説）

- **`output_config.effort` で思考の深さを制御。** `"high"` は複雑・曖昧なチケット（例: TKT-1046 の断続的500エラー）に深く推論させ、`"low"` は単純なチケット（重複請求・パスワードリセット）の推論を短く保ちます。500件/日のトリアージで、簡単／難しいを effort で振り分けるという運用設計につながります。
- **`format` は最終コールにだけ。** Phase 1 は `output_config={"effort": effort}` のみ（format 無し）。Phase 2 で `output_config={"effort": effort, "format": RESOLUTION_SCHEMA}` と **effort と format を同時指定** し、思考量を保ったまま構造化出力を得ます。
- **thinking ブロックの可視化。** ループ中に `block.type == "thinking"` を表示することで、「なぜエスカレーションしたか」「どの KB 記事を重視したか」という意思決定過程を監査できます（`block.thinking` にテキストが入る）。
- **`thinking={"type": "adaptive"}` と `effort` の関係。** adaptive 思考を有効化したうえで、`effort` がその「上限の深さ／使う度合い」を調整します。両方を毎回渡します。

### (c) つまずきやすい点

- `effort` は `thinking={...}` 側ではなく **`output_config` 側** に入れる（`output_config={"effort": effort}`）。
- Phase 2 で `effort` を落とさず、`format` と一緒に `output_config` へ両方入れる。
- thinking ブロックの中身は `block.thinking`（`block.text` ではない）。text ブロックは `block.text`。
- 高 effort はトークン消費・レイテンシが増える。`max_tokens=32000` と長めの `timeout` が効いてくる。

## 穴埋め5: ストリーミング版（`run_agent_streaming`）

### (a) 完成コード

```python
def run_agent_streaming(user_message: str, effort: str = "high") -> dict:
    """Run agent with streaming output."""
    messages = [{"role": "user", "content": user_message}]

    # --- Phase 1: streamed tool loop (effort only, NO format) ---
    while True:
        with client.messages.stream(
            model=MODEL,
            max_tokens=32000,
            system=SYSTEM_PROMPT,
            tools=tools,
            thinking={"type": "adaptive"},
            output_config={"effort": effort},
            messages=messages,
        ) as stream:
            for event in stream:
                if event.type == "content_block_start":
                    block = event.content_block
                    if block.type == "thinking":
                        print("\n💭 ", end="", flush=True)
                    elif block.type == "tool_use":
                        print(f"\n🔧 {block.name}: ", end="", flush=True)
                    elif block.type == "text":
                        print("\n📝 ", end="", flush=True)
                elif event.type == "content_block_delta":
                    delta = event.delta
                    if delta.type == "thinking_delta":
                        print(delta.thinking, end="", flush=True)
                    elif delta.type == "text_delta":
                        print(delta.text, end="", flush=True)
                    elif delta.type == "input_json_delta":
                        print(delta.partial_json, end="", flush=True)

        response = stream.get_final_message()

        if response.stop_reason != "tool_use":
            break

        # Execute tools and continue the loop
        messages.append({"role": "assistant", "content": response.content})
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                result = execute_tool(block.name, block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result,
                })
        messages.append({"role": "user", "content": tool_results})

    # --- Phase 2: final streamed call, constrained to the schema ---
    messages.append({"role": "assistant", "content": response.content})
    messages.append({
        "role": "user",
        "content": "Provide your structured resolution as JSON.",
    })

    print("\n\n📦 Structured resolution:\n", end="", flush=True)
    with client.messages.stream(
        model=MODEL,
        max_tokens=32000,
        system=SYSTEM_PROMPT,
        tools=tools,
        thinking={"type": "adaptive"},
        output_config={"effort": effort, "format": RESOLUTION_SCHEMA},
        tool_choice={"type": "none"},
        messages=messages,
    ) as stream:
        for event in stream:
            if event.type == "content_block_delta" and event.delta.type == "text_delta":
                print(event.delta.text, end="", flush=True)

    final = stream.get_final_message()
    return get_structured_result(final)
```

### (b) なぜそうするか（要点解説）

- **`stream()` を context manager で使う。** `with client.messages.stream(...) as stream:` とし、`for event in stream:` でイベントを逐次処理します。ダッシュボードに「今どのチケットを見ているか／何を検索しているか／どう推論しているか」をリアルタイム表示できます。
- **主要イベントの処理。**
  - `content_block_start`: 新しいブロックの開始。`event.content_block.type` で `thinking` / `tool_use` / `text` を見分け、見出し（💭/🔧/📝）を出す。
  - `content_block_delta`: トークンの増分。`delta.type` で分岐し、`thinking_delta`→`delta.thinking`、`text_delta`→`delta.text`、`input_json_delta`→`delta.partial_json` を流す。
- **`stream.get_final_message()`。** ストリーム終了後にこれを呼ぶと、完全な `Message` オブジェクト（全 content ブロック・`stop_reason`）が得られます。ループ継続判定（`stop_reason == "tool_use"`）と最終 JSON のパースはこの戻り値で行います。
- **2フェーズ構成は他関数と同じ。** Phase 1 は `effort` のみでツールループ、Phase 2 は `effort` + `format` + `tool_choice={"type": "none"}` で構造化出力をストリーミング。最終結果は `get_structured_result()` でパース。
- **`thinking={"type": "adaptive"}` を `stream()` にも渡す。** create と同様、ストリーミングでもアダプティブ思考を有効化します。

### (c) つまずきやすい点

- **delta の属性名を取り違える。** `thinking_delta`→`delta.thinking`、`text_delta`→`delta.text`、`input_json_delta`→`delta.partial_json`。text の delta から `.thinking` を読もうとしないこと。
- **`get_final_message()` は `with` ブロックの後（ストリーム消費後）に呼ぶ。** イベントを最後まで回しきってから取得します。
- **Phase 1 で `format` を付けない。** ストリーミングでも「ループ中はフォーマット無し」の原則は同じ。
- **無限ループ防止。** `while True:` を使う場合、`stop_reason != "tool_use"` で必ず `break` する。ツール結果を append し忘れると、同じターンを延々と繰り返します。
- **`input_json_delta` は部分 JSON。** 逐次表示用であり、完成した引数は `get_final_message()` の `tool_use` ブロックの `block.input` から取得します（途中の `partial_json` を自前で結合してパースしない）。

---

## まとめ

| パート | 追加した API 機能 | 鍵となる指定 |
|---|---|---|
| 1. ツールスキーマ | tool use | `name` / `description` / `input_schema`、`status` enum |
| 2. エージェンティックループ | tool use ループ＋adaptive thinking | `while stop_reason == "tool_use"`、thinking 含む全 content 返送 |
| 3. 構造化出力 | json_schema 出力 | 最終コールのみ `output_config={"format": ...}` ＋ `tool_choice={"type": "none"}` |
| 4. 思考量制御 | effort 制御 | `output_config={"effort": effort}`（最終は `format` も同梱） |
| 5. ストリーミング | streaming | `client.messages.stream()`、delta 種別で分岐、`get_final_message()` |

全パートを通じての一貫ルール:

1. **thinking ブロックは加工せず丸ごと返送する。**
2. **`format` 制約はツールループ中には付けず、最後の1回だけ付ける。**
3. **`thinking={"type": "adaptive"}` は毎回のリクエストに付ける。**
