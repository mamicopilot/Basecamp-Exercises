# 受講ノート — Prompt Rescue

壊れた本番プロンプトを「体系的に診断 → 根本原因を直す → eval で計測しながら反復改善」する練習。題材はサポートチケット処理（優先度分類・エンティティ抽出・返信ドラフト・JSON 出力）。

---

## ねらい / キーコンセプト

- **系統的なプロンプトデバッグ（systematic prompt debugging）**: 当てずっぽうで文言をいじらず、失敗ケースを観察 → 仮説 → 1変更 → 再計測のループで直す。
- **根本原因分析（root cause analysis）**: 症状（誤分類・捏造・JSON 破損）から、プロンプトのどの指示が原因かを切り分ける。
- **eval 駆動の反復（eval-driven iteration）**: 主観でなく、固定テストケース集合に対する合格率で良し悪しを判断する。
- **JSON 出力の堅牢化（robust structured output）**: キー・型・enum を固定し、地の文を排し、本文に特殊文字を混ぜない。
- **ハルシネーション抑制（hallucination mitigation）**: 「入力に明示された値だけを抽出。無ければ null」を徹底する。
- **採点基準に合わせて設計する**: プロンプトは「人間にとって良い文章」ではなく「採点関数を通す出力」を狙って書く。

## プロンプト改善チェックリスト

- [ ] 採点ロジック（何が PASS 条件か）を**先に**読んだか。
- [ ] 出力は JSON オブジェクト1個のみ。前後に説明・挨拶・コードフェンスを付けない指示があるか。
- [ ] 必須キーを固定したか（top: `priority` / `entities` / `response` / `confidence`、entity: `product` / `version` / `error_codes` / `affected_users`）。
- [ ] enum を厳密化したか（`priority` は `P1`〜`P4`、`confidence` は `high`/`medium`/`low`、装飾なし）。
- [ ] 型を固定したか（`error_codes` は配列、無ければ `[]`／`affected_users` は文字列か null）。
- [ ] 優先度ルールが**人数依存・トーン依存**になっていないか。深刻度（セキュリティ/PII・データ損失・全停止・支払い/給与ブロック）を人数より優先しているか。
- [ ] 「複数問題は最も重い1件で分類」と明記したか（平均しない）。
- [ ] 「入力に無い値は推測せず null」「曖昧表現から数を作らない」と明記したか。
- [ ] 原文の文字（`ñ` などアクセント/非ラテン文字）を改変しない指示があるか。
- [ ] 返信作法（P4 に修正を約束しない／曖昧なら確認質問／複数問題は全件言及／常にプロフェッショナル）を入れたか。
- [ ] 変更は一度に1種類にして、再計測で因果を確認したか。

## よくある落とし穴 / ベストプラクティス

**落とし穴**
- 緊急語・全文大文字・感情に引きずられて優先度を上げ下げする（id 11 は P4、id 17 は P1）。
- 「few users → P3」の機械適用でセキュリティ/PII 案件を取りこぼす（id 19 は1名でも P1）。
- 複数問題を平均して中間値に丸める（id 5 は P1、id 6 は P2）。
- 情報が無いのに製品名・人数を捏造する（id 7・id 21・id 18）。null は減点されないのに、埋めようとして失格する。
- JSON の前後に "Here is the analysis:" などの地の文を付ける／`response` に改行・コードブロック・生引用符を入れて壊す。
- アクセント記号や非ラテン文字を勝手に正規化して別物にする（id 20 の `Ñoño Analytics`）。

**ベストプラクティス**
- 「分からない/書いていない＝ null（または `[]`）」を既定動作にする。**捏造は失格、null は安全**。
- 優先度ルールは「上から順に評価」の意思決定リストにして、最初に当てはまった段階で確定させる。
- 出力契約（キー・型・enum・地の文禁止）をプロンプト冒頭で明文化する。
- 変更 → `run_eval_suite` → カテゴリ別バーと失敗理由を確認、を小刻みに回す。
- 迷ったら `compare_prompts` で A/B、`show_iteration_log` で推移を見る。

## 持ち帰りポイント

1. **採点基準を先に読む。** プロンプトは「読みやすい文章」ではなく「採点関数を通る出力」を狙って設計する。
2. **優先度は内容で決める。** トーン・緊急語・人数ではなく、セキュリティ/PII・データ損失・全停止・支払い/給与ブロックといった実害を最優先で評価し、複数問題は最も重い1件で分類する。
3. **null は味方、捏造は敵。** 入力に明示が無ければ素直に null/空配列にする。曖昧表現から数値を作らない。
4. **JSON はオブジェクト1個だけ。** 地の文・コードフェンス・本文の特殊文字を排し、キー・型・enum を固定する。原文の文字は改変しない。
5. **一度に1変更して計測する。** `run_eval_suite` → `compare_prompts` → `show_iteration_log` で、推測ではなくスコアの差分で改善を裏付ける。
