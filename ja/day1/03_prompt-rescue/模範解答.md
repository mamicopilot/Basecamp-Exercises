## 0. 評価基準（採点ハーネス）を先に理解する

プロンプトを直す前に、「何で点が付くか」を読み解くことが最重要です。プロンプトは採点ロジックに**厳密に合わせて**設計します。`Prompt_Rescue_solo.py` の採点モジュールから読み取れる事実は次のとおりです。

採点は 1 ケースにつき 4 つの基準すべてを満たして初めて PASS（`score_case`）。

| 基準 | 関数 | 合格条件（要点） |
|---|---|---|
| `json_valid` | `check_valid_json` / `parse_output` | トップレベルに `priority` `entities` `response` `confidence` の4キー、`entities` 内に `product` `version` `error_codes` `affected_users` の4キーが**すべて**存在し、`entities` がオブジェクトであること。出力中の最初の波括弧ブロックが JSON として解釈される（前後の地の文があっても抽出は試みるが、不正確なら失敗）。 |
| `priority_correct` | `check_priority` | `priority` を大文字化した値が gold と**完全一致**（例: `P1`）。`"P1 - high"` のような装飾は不一致になる。 |
| `entities_accurate` | `check_entities` / `_value_in_input` | 空でない各エンティティ値が**入力テキストから導出可能（ほぼ部分一致）**であること。導出不能＝ハルシネーションとして失格。**null / 空文字 / 空配列は常に合格**。 |
| `response_coherent` | `judge_response`（別 LLM 判定） | `audited_response: true` のケース（id 5, 8, 11, 17, 19）のみ判定。非監査ケースは自動 PASS。 |

ここから導かれる中心戦略は明快です。

- **優先度は内容ベースで厳格なルール化**し、enum 値を `P1`〜`P4` の4つだけに限定する。
- **エンティティは「入力に明示されたものだけ」を抜き、それ以外は迷わず null/空にする**（推測は禁止。null は減点されないが、捏造は即失格）。
- **出力は JSON オブジェクト1個だけ**にし、キーと型を固定する。

## 1. 故障分析（元プロンプトの欠陥）

元の `system_prompt`（抜粋）:

```text
You are a support ticket processor. For each ticket, you must:
1. Classify priority (P1-P4) based on business impact
2. Extract: product name, version, error codes, affected users count
3. Draft a helpful response ...
4. Return everything as JSON: {"priority": "", "entities": {...}, "response": "", "confidence": "high/medium/low"}

Rules:
- P1 = system down, all users affected
- P2 = major feature broken, many users affected
- P3 = minor bug, few users affected
- P4 = feature request or cosmetic issue
- If unsure about priority, use your best judgment
- Response should be professional and empathetic
- Always include all JSON fields even if empty
- Be concise but thorough
```

### 欠陥A: 優先度ルールが「ユーザー数」と「トーン」に引きずられる

P1〜P4 の定義が "all/many/few users" という**ユーザー数依存**で書かれており、深刻度の本質（システム停止・データ損失・セキュリティ/PII 漏えい・支払い/給与ブロック）が反映されていません。「If unsure, use your best judgment」という指示が**感情やトーンに基づく当てずっぽう**を許してしまいます。

- **id 19（UserVault PII 漏えい）**: 報告者は 1 名だが gold は **P1**。「few users → P3」の定義に従うと誤って P3 に落とす。→ セキュリティ/PII は人数に関わらず P1、というルールが欠落。
- **id 17（全文大文字の激怒チケット）**: gold は **P1**（システムクラッシュ＋データ損失＋150名ブロック）。一方で **id 11（"CRITICAL"/"IMMEDIATELY" を連呼するが実体は SSO 未対応の機能要望）** は gold **P4**。元プロンプトは緊急語・大文字に反応して両方を P1/P2 に寄せがち。→ **優先度はトーンではなく内容で決める**ルールが欠落。
- **id 5（請求バグ＋エクスポート不可、給与ブロック）**: gold **P1**。複数問題を「平均」して P2 に丸めやすい。→ 複数問題は**最も重い方**で分類するルールが欠落。
- **id 6（ダッシュボードのバグ＋ダークモード要望）**: gold **P2**。バグ(P2)と要望(P4)を平均して P3 に丸めやすい。→ 同上。

### 欠陥B: エンティティのハルシネーション

「Extract: product name, version, ...」としか書かれておらず、**情報が無いときにどうするか**の指示がありません。モデルは空欄を嫌って値を捏造します。採点では捏造は即失格（`check_entities`）。

- **id 7（"a bunch of stuff seems off"）**: product は null が正解。ノート曰く「`PageLoader` のような製品名を幻覚する」典型失敗。
- **id 21（"several people" / "other departments"）**: `affected_users` は **null** が正解。「several」から `5` や `10` を捏造すると失格。
- **id 18（"it's broken. please fix."）**: 全エンティティ null/空が正解。製品・優先度を自信満々に当てると失敗。
- **id 3（ツールチップ切れ）/ id 16（フォローアップ）**: `product` は **null** が正解（製品名が明示されていない）。一般名詞から製品名をでっち上げないこと。

### 欠陥C: JSON 出力が壊れやすい

- 「Be concise but thorough」「Draft a helpful response」が、**JSON の前後に地の文（"Here is the analysis:" 等）**を付けたり、`response` 内の改行・引用符をエスケープし損ねたりする原因になります。`parse_output` は波括弧ブロックを拾おうとしますが、`response` が長文・コードスニペット・特殊文字を含むと壊れやすい。
- **id 10（非ネイティブ英語）**: ノート曰く「不自然な言い回しで JSON が壊れる」典型。
- **id 13（200語超の長文）/ id 14（スタックトレース付き）**: フィールド切り詰めや、JSON 内コードブロックの混入で破損しやすい。
- **id 20（製品名に `ñ`、報告者名にアクセント記号）**: gold product は **`Ñoño Analytics`**。`ñ` を `Nono`/`Noo` に化けさせると失格。→ **入力の文字を改変せず原文ママで抽出**する指示が必要。

### 欠陥D: enum・キー・型が緩い

`confidence` は `"high/medium/low"` と例示はあるが厳密化されておらず、`priority` も装飾付き（`"P1 (high)"`）になりがち。`error_codes` は配列のはずが文字列で返ることがある。`affected_users` の型（文字列か数値か）も曖昧。**キー名・型・enum を固定**しないと `check_valid_json` / `check_priority` を落とす。

### 欠陥E: 監査対象レスポンスの作法が不足

`audited_response: true` のケースは別 LLM（`JUDGE_SYSTEM_PROMPT`）が回答文を採点します。判定基準は「分類と矛盾しない／本文の内容に触れる／**P4 機能要望に "fix" を約束しない**／曖昧チケットでは**追加情報を求める**／複数問題は**全部に言及**／トーンは常にプロフェッショナル」。元プロンプトの「helpful, empathetic」だけではこれらを満たしません。

- **id 11 / id 4**: 機能要望に「修正します」と言うと FAIL。
- **id 8 / id 18（曖昧）**: 確認質問が無いと FAIL。
- **id 5（複数問題）**: 片方（エクスポート）にしか触れないと FAIL。

## 2. 修正方針（欠陥 → 対処）

| 欠陥 | 対処方針 |
|---|---|
| A. 優先度の誤分類 | 優先度を**意思決定の上から順に評価する明確なルール**に書き換える。(1) セキュリティ/プライバシー/PII 漏えい、データ損失・破損、システム全停止、支払い・給与のブロック → 人数によらず **P1**。(2) 主要機能の障害 → **P2**。(3) 軽微・断続・回避策あり・情報不足 → **P3**。(4) 機能要望・見た目だけ → **P4**。さらに「**優先度はトーン/緊急語/大文字ではなく内容で決める**」「**複数問題は最も重い1件で分類**」を明記。 |
| B. ハルシネーション | 「**入力に明示された値だけ**を抽出。推測・常識補完・概数の数値化を禁止。明示が無ければ `null`（配列は `[]`）」。「several / a few / some など曖昧表現から人数を作らない」を明記。 |
| C. JSON 破損 | 「**JSON オブジェクトを1個だけ出力。前後に地の文・コードフェンス・説明を一切付けない**」。`response` は**1段落のプレーンテキスト**に限定し、改行・コードブロック・生引用符を避ける指示を入れる。 |
| D. enum/キー/型 | キー名・順序・型を固定。`priority` は `P1`/`P2`/`P3`/`P4` の4値のみ（装飾禁止）。`confidence` は `high`/`medium`/`low` のみ。`error_codes` は文字列配列、`affected_users` は文字列または null。**全フィールド常時出力**。 |
| E. 監査レスポンス | 「P4 機能要望には修正を約束せず要望として受領」「情報不足なら必ず具体的な確認質問」「複数問題は全件に言及」「相手のトーンに関わらずプロフェッショナル」を回答ルールに明記。文字に依存しないよう原文の文字（`ñ` 等）はそのまま使う。 |

> 補足: `confidence` の値自体は採点（`check_valid_json`）ではキー存在のみが見られ、値の中身は機械採点では検査されません。ただし監査ケースの judge は「曖昧なら情報を求める」回答を期待するため、**情報が薄いケースで `confidence: low` にし、回答で確認質問をする**のが整合的です。

## 3. 改良版プロンプト全文（完成形）

そのまま `system_prompt` に貼り替えて `run_eval_suite(system_prompt, label="Fixed v1")` を実行する想定の完成形です。出力 JSON のキー・enum・型は**原典の採点基準に厳密一致**させています（本文は日本語、構造値は英語のまま）。

```text
あなたはサポートチケット処理エンジンです。入力されたチケット1件を分析し、結果を「JSONオブジェクト1個」だけで返します。

# 出力契約（厳守）
- 出力はJSONオブジェクト1個のみ。前後に説明文・挨拶・```などのコードフェンスを一切付けない。
- キー名・構造・型は以下に厳密一致させる。全フィールドを常に含める（値が無くてもキーは省略しない）。

{
  "priority": "P1 | P2 | P3 | P4 のいずれか1つ",
  "entities": {
    "product": "製品名の文字列、または null",
    "version": "バージョン文字列、または null",
    "error_codes": ["エラーコード文字列の配列。無ければ空配列 []"],
    "affected_users": "影響ユーザー数の文字列、または null"
  },
  "response": "顧客への返信（プレーンテキスト1段落）",
  "confidence": "high | medium | low のいずれか1つ"
}

- priority の値は厳密に "P1" "P2" "P3" "P4" の4つだけ。"P1 (high)" のような装飾・補足は禁止。
- confidence の値は厳密に "high" "medium" "low" の3つだけ。
- response は改行・コードブロック・生の二重引用符を避けた1段落のプレーンテキストにする（JSONが壊れないように）。

# 優先度の判定ルール（上から順に評価し、最初に当てはまった段階で確定する）
重要: 優先度は「文章のトーン・緊急語・大文字・感情」ではなく「実際の業務影響（内容）」だけで決める。

1. 次のいずれかに該当すれば、報告者の人数に関係なく P1：
   - セキュリティ／プライバシーの問題、他ユーザーの個人情報(PII)漏えい
   - データの損失・破損
   - システム全体のダウン／ログイン不可など全面停止
   - 支払い・給与・請求処理など、重大な業務プロセスのブロック
2. 主要機能が壊れている、または大きく劣化している（多くの業務が滞るが全停止ではない）→ P2
3. 軽微なバグ、断続的な問題、回避策がある、情報が乏しく深刻度を確定できない → P3
4. 機能追加の要望、見た目だけの問題、改善提案 → P4（"今すぐ" "CRITICAL" など強い言葉があってもP4のまま）

複数の問題が1通に含まれる場合は、最も重い1件に基づいて全体の優先度を決める（平均や折衷をしない）。

# エンティティ抽出ルール（ハルシネーション厳禁）
- 入力テキストに明示されている値だけを抽出する。推測・常識による補完・言い換えをしない。
- 明示が無いフィールドは product/version/affected_users なら null、error_codes なら [] にする。
- 製品名・バージョン・エラーコードは入力に書かれた文字をそのまま使う（アクセント記号や非ラテン文字 例: ñ も改変しない）。
- affected_users は具体的な数値が書かれている場合のみ記入する。"several" "a few" "some" "many" "他部署も" のような曖昧表現から数を作らない（その場合は null）。
- error_codes はチケット本文や貼り付けられたエラー応答・スタックトレースに現れる識別子（例: SVC-503-AUTH, 500, 429, SYNC_FAILED, "timeout exceeded" など）を配列で列挙する。無ければ [] 。
- 複数の製品名が出てくる場合は、ユーザーが実際に使用・契約している主たる製品を product にする。

# 返信(response)の作法
- 常にプロフェッショナルかつ共感的に。相手が攻撃的・全文大文字でも冷静で丁寧に保つ。
- 割り当てた優先度と矛盾しないこと。
- 機能要望(P4)には「不具合として修正します」と約束しない。要望として受領し、検討プロセスや今後の連絡について期待値を伝える。
- 情報が不足している曖昧なチケットでは、具体的な確認質問（影響を受けた機能・エラーメッセージ・人数・発生時刻など）を必ず1つ以上含める。
- 1通に複数の問題がある場合は、すべての問題に言及する（片方だけにしない）。

# confidence の付け方
- 製品・バージョン・エラー・人数などの根拠が揃い分類が明確 → high
- 一部のみ判明 → medium
- 情報が乏しく推測の余地が大きい → low
```

> メモ: 上記は**単一プロンプト**での完成形です。ハーネスは `run_eval_suite([step1, step2, step3], ...)` のチェーン実行にも対応しますが、本課題の難所（人数に依存しない優先度・捏造禁止・JSON 厳格化）は単一プロンプトのルール化で十分カバーでき、API 呼び出しコストも 1/3 で済みます。チェーン化は「分類」「抽出」「回答生成＋組み立て」に分ける設計が有効ですが、最終ステップで上記の出力契約を必ず再掲してください。

## 4. 反復のコツ（eval harness の使い方）

1. **まずベースラインを取る**: `results = run_eval_suite(system_prompt, label="Baseline (Broken Prompt)")`。最初の実行が自動的にベースライン基準（`_baseline_score`）に登録され、以降の実行は差分（`[+N]`）付きで表示される。
2. **カテゴリ別バーで弱点を特定**: 結果の「Score by Category」で、`vague` / `complex` / `multi_issue` が落ちているはず。失敗ログには `priority_correct: Priority=P3, expected=P1` のように**どの基準が落ちたか**が出るので、欠陥A/B/C のどれかを切り分ける。
3. **一度に1種類の修正**: 「優先度ルールだけ直す」→ 再実行 → 差分確認、のように変更を小さく保つと因果が分かる。複数同時に変えるとどれが効いたか不明になる。
4. **A/B で比較**: 迷ったら `compare_prompts(prompt_a, prompt_b, label_a="Current Best", label_b="New Variant")` で同一ケース集合に対し横並び比較。カテゴリ別 Delta 表が出る。
5. **進捗を俯瞰**: `show_iteration_log()` でスコア推移の折れ線（合格ライン 75%、最適化ティア 90%）を確認。
6. **ハルシネーション失敗の読み方**: `entities_accurate` が `Possible hallucinated entities: affected_users="5"` のように落ちたら、その値が入力に無いのに出している合図。該当フィールドを null/空にできているか確認する（null は常に合格）。
7. **監査ケース（5,8,11,17,19）**: ここは別 LLM judge が回答文も見る。落ちたら回答の作法（P4で修正を約束した／曖昧なのに質問していない／複数問題の片方を無視した、等）を疑う。
