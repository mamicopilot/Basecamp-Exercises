# 受講ノート — Diagnosing AI Problems

## ねらい
- 顧客エスカレーション（Priya のメール）を起点に、**コードを動かさず**、成果物だけから AI システムの障害を診断する練習。
- 学ぶ中心スキルは、エージェンティックシステムの 3 構成要素を「読める」ようになること: **システムプロンプト・ツール定義・実行トレース**。
- 進め方（README）: ペア/3 人で。(1) メールだけで 10 分、仮説を 3 つ。(2) 成果物 20 分、各仮説を**行レベルの証拠**で確認/反証。

## キーコンセプト — エージェンティックシステムの読み方
| 構成要素 | 何を読むか | この演習での例 |
|---|---|---|
| システムプロンプト | エージェントの役割・手順・分類基準・統合の責務 | コーディネーターのプロセス／分類ガイド／「読みやすくするだけ」という統合定義 |
| ツール定義（JSON） | モデルから見える名前・説明・引数。**いつ呼ぶか**が説明から分かるか | `fetch_customer_v2_databricks` の用途欠落、重複・ゴミツール、残存 TODO メモ |
| 実行トレース | 実際の手順・ツール呼び出し・結果・誤呼び出し/ループ・usage | 顧客特定で 3 ツール＋1 エラー、specialist 1 体しか起動せず、cache 0/0 |

**合言葉**: 「プロンプトがおかしそう」は証拠ではない。「**何行目に何と書いてある／トレースの toolu_XX でこう呼んでこう失敗した**」が証拠。

## 診断フレームワーク要点（手順カード）
`diagnostic-framework.md` の 4 ステップ。各ステップの「出力」を必ず残すこと。

1. **症状 (Symptom)** — 顧客の言葉のまま。解釈を足さない。→ 出力: 平易な問題説明。
2. **仮説 (Hypothesis)** — 成果物を**見る前**に最低 3 つ。立場を先に明確化し、自分の診断の勘がどこで強い/弱いか学ぶ。→ 出力: ランク付き仮説リスト。
3. **証拠 (Evidence)** — 各成果物で各仮説を確認/反証。**行を引用**。最初に当たる 3 点: ①システムプロンプト（何をしようとしているか）②ツール説明（いつ呼ぶか分かるか）③トレース（実際に何が起きたか）。→ 出力: 仮説ごとの行引用付き証拠。
4. **推奨 (Recommendation)** — 診断した根本原因に限定した修正。「どのファイルの・どの行を・どう変え・なぜその障害を防ぐか」。書けないならステップ 3 に戻る。→ 出力: 根本原因ごとの限定修正。

## よくある障害モードと兆候
| 障害モード | トレース/成果物に出る兆候 |
|---|---|
| ルーティング/分類の失敗 | 複合チケットなのに specialist を 1 体しか起動しない。分類が単一値の enum。「2 つに当てはまれば片方を選べ」という指示 |
| ツール説明が曖昧で安定して呼べない | description に**いつ呼ぶか（トリガー言語）**が無い。実装名（v2/Databricks）だけ。重複ツール乱立で区別不能。トレースで当てずっぽう→エラー連鎖 |
| エスカレーションパスの欠如/未使用 | `escalate_to_human` 等が定義されているのにトレースで一度も呼ばれない。直せない件を顧客に丸投げ |
| サブエージェントの過剰申告 | 権限/ツールの無い作業を「対応済み」と報告。コーディネーターが無検証で `resolved` 化 |
| コンテキストが必要なモデルに届かない | 上流が取得した情報を下流に渡さず、下流が読み上げ回答に終始（メトリクス未使用） |
| キャッシュ配置が接頭辞を壊す | 接頭辞先頭に毎回変わる値（timestamp/request id）。usage の cache_creation / cache_read がともに 0 |

## Claude を「オラクルではなくプロファイラ」として使う
README の指示: 根本原因を断定する前に Claude に貼り、**答えとしてではなく評価対象の仮説として**扱う。

- メールだけの段階の問い（仮説生成）:
  > 「失敗中のマルチエージェント・サポートシステムについての顧客メールです。最も可能性の高い**構造的な**根本原因は？ 3 つの仮説と、それぞれ成果物の**どこを見れば確認できるか**を 1 つずつ。」
- 成果物の段階の問い（プロファイル）:
  > 「コーディネーターのツール説明です:【貼り付け】。この説明だけでモデルは**いつこのツールを呼ぶか**分かりますか？ 何が欠けていますか？」

ポイント: Claude に「結論」を求めず、「**どこを見ればよいか／何が欠けているか**」を聞き、自分でトレースの行に当てて検証する。

## 持ち帰りポイント
1. **証拠は行で示す**。ファイル名＋行番号／トレースの `toolu_XX` を引いて初めて「診断」。印象は仮説止まり。
2. **単一ディスパッチは複合チケットを必ず落とす**。enum とプロンプトの「片方を選べ」を疑う。
3. **ツール説明は『いつ呼ぶか』が命**。実装名だけ・重複・TODO 残しはモデルの判断を壊す。リネーム＋トリガー言語＋統廃合。
4. **コーディネーターは清書係ではなく検証者**。サブエージェントの過剰申告を鵜呑みにせず、未解決はエスカレーション。`resolved` は実際に解決してから。
5. **キャッシュは接頭辞の先頭から効く**。timestamp / request id など揮発値を先頭に置かない。usage の `cache_read_input_tokens` で効果を確認。
