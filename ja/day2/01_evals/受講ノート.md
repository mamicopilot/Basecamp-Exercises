# 受講ノート — Evals

AI ショッピングアシスタント `boutique` を題材に、エージェント向け eval（評価スイート）を「ゼロから組んで、壊れる所を数値で見つけ、直して再計測する」流れを学ぶセクションのノートです。

---

## ねらい

- 「試して様子を見る（vibes）」から、**数値で示せる再現可能なフィードバックループ**へ移行する。
- eval が与えてくれる 3 つの価値を体得する。
  - **ベースライン**: 今このエージェントはどのくらい良いか（クエリ種別ごとに）。
  - **フィードバックループ**: プロンプト/ツール設計/モデルを変えたら、本当に良くなったか・回帰がないかを再実行で確認。
  - **自信**: 顧客にリリースする前に、感覚ではなく**数字**を示せる。

---

## キーコンセプト

eval は次の 4 部品で構成される。

```
Tasks ──> Runner ──> Graders ──> Results
```

- **テストタスク定義（Tasks）**: 「何を」テストするか。`id` / `description` / `query` / `category` / `graders` を持つ dict。能力（capability）とエッジケースを網羅するよう選ぶ。
- **Grader**: 「正しく振る舞えたか」を 0/1 で採点し、理由（reason）を返す関数。`(result, check, context=None) -> {"score", "reason"}` の共通インターフェース。
- **Harness（Runner）**: クエリをエージェントへ送り、トランスクリプトを集め、grader を適用して結果を構造化する（`run_eval`）。表示（`print_summary`）・保存（`save_results`）・個別調査（`inspect_task`）は分離。
- **eval 駆動の反復**: 失敗を根拠に「grader を緩める」のではなく「**エージェントを直す**」。直したら再実行して効果と回帰を数値で確認。

**重要ルール**: 1 タスクが pass するのは「**全 grader の全 check が 1.0**」のときだけ（`all(g["score"] == 1.0 ...)`）。grader が空のタスクは pass しない。

**失敗（failure）とエラー（error）は別物**: エージェントが例外を投げたら *failed run*（`error` フィールドに記録、採点はスキップ）＝インフラが壊れた。間違った答えを返したら *failed task*（grader が普通に走り何が悪いか報告）＝エージェントの中身が悪い。

## eval 設計チートシート

### grader の型と check 形式

| grader | 何を見るか | check の形式 | 使いどころ |
|--------|-----------|-------------|-----------|
| `response_contains` | 最終応答に文字列が含まれるか（大文字小文字無視） | `"49.99"` / `"sneakers"` | 一意のキーワード/価格の含有 |
| `response_numeric` | 最終応答に許容差内の数値が含まれるか | `{"value": 139.95, "tolerance": 0.10}` | 合計・割引など数値の正解（丸め差を tolerance で吸収） |
| `tool_use` | 特定ツールを（指定引数で）呼んだか | `{"tool_name": "get_product"}` / `{"tool_name": "get_product", "arguments": {"product": "t-shirt"}}` | 答えを**ツールに接地**したか（捏造・暗算でないか）。引数は部分一致（指定キーのみ照合） |
| `llm_judge` | 自由応答が自然言語の基準を満たすか | `"Response describes the available products"` | 正解が一意でないオープンエンド。基準は**1 つずつ atomic**に |

### grader 選択の指針

- **正解が一意 → 決定的 grader**（完全一致 `response_contains` / 数値許容 `response_numeric` / ツール接地 `tool_use`）。安い・速い・再現可能。
- **正解が複数あり得る → LLM-as-judge**。意味的合致やトーンを判定。`temperature=0.0`、1 行目に `PASS`/`FAIL` を強制して解析を安定化。
- **`tool_use` の引数指定は最小限に**。「その引数値こそテスト対象」のときだけ `arguments` を付ける（例: `t-shirt` のハイフン）。順序や回数は縛らない。

## よくある落とし穴

- **grader を緩めて「合格」にする**。原因はエージェント側にあることが多い。grader ではなくプロンプト/ツール/実装を直す。
- **`tool_use` を厳しく縛りすぎる**。呼び出し順序や全引数を固定すると、正しい別経路を不当に FAIL させる。「接地できたか」を見るに留める。
- **LLM judge に複数基準を詰め込む**。1 呼び出し 1 基準（atomic）。混ぜると採点が曖昧になり、どこで落ちたか分からない。
- **1 回の実行で結論づける**。LLM は temperature 0 でも非決定的。`num_runs` を増やしてベースラインを取り、改善後も複数回で回帰確認。
- **judge の出力を自由文のまま解析**。`PASS`/`FAIL` を強制し、解析不能時は黙って成功にせず 0.0＋理由を返す。
- **error と failure を混同**。API 失敗（error）を「不正解（failure）」として集計すると指標が汚れる。

## ベストプラクティス

- **Registry パターン**: grader は `GRADER_REGISTRY` の dict で type 名から引く。新 grader は「関数 1 つ＋dict 1 行」で追加でき、分岐チェーンを触らない。
- **Runner は表示しない**: `run_eval` はデータ構造を返し、表示/保存/調査は別関数。差し替え・ダッシュボード連携が容易。
- **エージェントは引数で渡す**: runner が `agent_fn` を受け取る。モック差し替え・別エージェント比較・計装が楽。
- **トランスクリプト解析は runner が持つ**: エージェントは生メッセージを返し、`final_text` / `tool_calls` 抽出は runner 側。エージェントは「どう評価されるか」を知らなくてよい。
- **カテゴリで集計**: `category` を付けて能力領域別の合格率を出し、弱点を特定。
- **結果を JSON 保存**: `save_results` で実行ごとに保存し、実行間で比較・回帰検出。

## 持ち帰りポイント

1. **eval は感覚を数値に変える装置**。ベースライン → 変更 → 再計測のループで「良くなった/回帰した」を証明できる。
2. **正解が一意なら決定的 grader、一意でないなら LLM-as-judge**。決定的を優先し（安く速く再現可能）、自由応答だけ judge に回す。
3. **grader は脆くしすぎない**。`tool_use` は「答えをツールに接地したか」を見る程度に。引数固定は「その値こそテスト対象」のときだけ。
4. **失敗の原因はエージェント側**。`boutique` は system プロンプト・tool specs・エラーハンドリングの 3 点改善で約 50% → 約 100% に。grader を緩めて通すのは本末転倒。
5. **非決定性を前提に運用する**。`num_runs` を増やしてベースラインを安定させ、改善後も複数回で回帰確認。error（インフラ障害）と failure（不正解）は分けて集計する。
