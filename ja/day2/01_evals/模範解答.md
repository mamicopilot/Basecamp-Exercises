## 全体像 — この演習で埋める「穴」

原典の穴埋め（TODO）は次の 4 系統です。本解答はこの順に節を立てます。

1. **Part 3: テストタスクの定義** — 5 つの例題クエリに対する task dict を完成させる（`tasks` リスト）。
2. **Part 6: LLM-as-judge grader の実装** — `grade_llm_judge(result, check, context=None)` を実装し、`GRADER_REGISTRY` に登録する。
3. **Part 6: LLM-as-judge を使うタスクの追加** — `llm_judge_tasks`（あるいは `tasks` への統合）を定義する。
4. **Part 5: エージェントの改善** — eval の失敗から原因を突き止め、system プロンプト / tool specs / ツール実装を改善する。

eval の構造は次の 1 行で表されます。

```
Tasks ──> Runner ──> Graders ──> Results
```

- **Tasks**: 何をテストするか（シナリオ・期待挙動・採点方法）
- **Runner**: `run_eval` がクエリをエージェントに送り、トランスクリプトを集め、grader を適用
- **Graders**: 正しく振る舞えたかを 0/1 で採点し、理由を返す
- **合否ルール**: 1 タスクは**全 grader の全 check が 1.0 のときのみ pass**（`passed = all(g["score"] == 1.0 for g in grades)`）。grader が空のタスクは pass しない仕様。

## 節 1. Part 3 — テストタスクの定義（穴埋め: `tasks` の 5 件）

### (a) 完成コード

原典 `tasks` リストの「Build tasks for these queries」コメント部分を、次の 5 件で埋めます（先頭の `price_jeans` は与えられた参照タスク）。

```python
tasks = [
    # ── Reference task ─────────────────────────────────────────────────────
    {
        "id": "price_jeans",
        "description": "Direct price lookup for jeans",
        "query": "How much do jeans cost?",
        "category": "product_lookup",
        "graders": [
            {"type": "response_contains", "checks": ["49.99"]},
            {"type": "tool_use", "checks": [{"tool_name": "get_product", "arguments": {"product": "jeans"}}]},
        ],
    },

    # ── 1. ハイフンを含む商品名（エッジケース）────────────────────────────
    # カタログのキーは "t-shirt"。エージェントが "t-shirt" を渡せば正解。
    # "tshirt" / "t shirt" を渡すと素の get_product では KeyError になる。
    # arguments 付き tool_use でキーの正確さを検証する。
    {
        "id": "price_tshirt",
        "description": "Price lookup with hyphenated product name",
        "query": "Price of a t-shirt?",
        "category": "product_lookup",
        "graders": [
            {"type": "response_contains", "checks": ["24.99"]},
            {"type": "tool_use", "checks": [{"tool_name": "get_product", "arguments": {"product": "t-shirt"}}]},
        ],
    },

    # ── 2. 同義語 / カタログ外（"shoes" は無い。"sneakers" が該当）────────
    # 素のエージェントでは FAIL する想定のタスク。価格を捏造せずツールを
    # 試したこと（tool_use）と、改善後に "sneakers" を代替提案できること
    # （response_contains）を確認する。
    {
        "id": "price_shoes_synonym",
        "description": "Synonym query: 'shoes' is not in catalog ('sneakers' is)",
        "query": "How much for shoes?",
        "category": "product_lookup",
        "graders": [
            {"type": "tool_use", "checks": [{"tool_name": "get_product"}]},
            {"type": "response_contains", "checks": ["sneakers"]},
        ],
    },

    # ── 3. 複数ツール（lookup × 2 + calculation）──────────────────────────
    # shirt = 29.99, belt = 24.99
    # 3 * 29.99 = 89.97, 2 * 24.99 = 49.98, 合計 = 139.95
    {
        "id": "total_shirts_belts",
        "description": "Multi-item total requiring product lookups + calculation",
        "query": "3 shirts and 2 belts, what's my total?",
        "category": "multi_tool",
        "graders": [
            {"type": "response_numeric", "checks": [{"value": 139.95, "tolerance": 0.10}]},
            {"type": "tool_use", "checks": [
                {"tool_name": "get_product"},
                {"tool_name": "calculate", "arguments": {"op": "*"}},
                {"tool_name": "calculate", "arguments": {"op": "+"}},
            ]},
        ],
    },

    # ── 4. パーセント計算（lookup + percentage math）──────────────────────
    # jacket = 89.99, 20% off = 89.99 * 0.80 = 71.992 → 値引後価格 71.99 を採点。
    # 許容差 0.10 で 71.99 / 71.992 / 72.00 の丸め差を吸収する。
    {
        "id": "discount_jacket",
        "description": "Calculate 20% off a jacket (lookup + percentage math)",
        "query": "What's 20% off a jacket?",
        "category": "calculation",
        "graders": [
            {"type": "response_numeric", "checks": [{"value": 71.99, "tolerance": 0.10}]},
            {"type": "tool_use", "checks": [
                {"tool_name": "get_product"},
                {"tool_name": "calculate"},
            ]},
        ],
    },

    # ── 5. オープンエンド（LLM-as-judge が必要。節 3 で追加）───────────────
    # "What do you sell?" は決定的 grader では採点しづらいため、節 3 の
    # llm_judge_tasks 側（what_do_you_sell）で扱う。
]
```

### (b) 設計意図

**テストタスクの選び方（capability × edge case のカバレッジ）**
5 つのクエリは「エージェントの能力をひととおり踏む × 壊れやすい所を突く」よう選ばれています。

| id | category | 狙い |
|----|----------|------|
| `price_jeans` | `product_lookup` | 基本の幸せパス（参照） |
| `price_tshirt` | `product_lookup` | ハイフン入りキーの**引数フォーマット**を試す |
| `price_shoes_synonym` | `product_lookup` | **カタログ外の同義語**処理（捏造しない／代替提案） |
| `total_shirts_belts` | `multi_tool` | **複数ツール連携**（lookup×2 + 乗算 + 加算） |
| `discount_jacket` | `calculation` | **パーセント計算**（lookup + 割引算） |

`category` を付けるのは `print_summary` が**カテゴリ別合格率**を出すためで、「どの能力領域が弱いか」を数値で示せます。

**決定的 grader と LLM-as-judge の使い分け（このパートでの選択）**
- 価格・合計・割引のように**正解が一意に決まる**ものは決定的 grader（`response_contains` / `response_numeric`）で採点。安く・速く・再現可能。
- 「正解の出し方が一意でない」`What do you sell?` は決定的 grader が向かないため、節 3 の LLM-as-judge に回す（原典の Part 6 ヒントどおり）。

**`tool_use` grader の脆さへの配慮（引数を指定する／しないの判断）**
原典の指針: tool_use は「答えを**ツールに接地**したか（捏造・暗算していないか）」を見るためのもので、**呼び出し順序を縛るものではない**。

- `price_tshirt` は「ハイフン入りキーを正しく渡せたか」が**まさにテスト対象**なので `arguments={"product": "t-shirt"}` を指定。
- `price_shoes_synonym` は「ツールを試したか」だけを見たいので `{"tool_name": "get_product"}`（引数なし）。`shoes` は存在しないキーなので、ここで引数を縛ると評価が不当になる。
- `total_shirts_belts` の `calculate` は `{"op": "*"}` / `{"op": "+"}` という**演算子だけ**を部分一致で確認（`grade_tool_use` は指定キーのみ部分照合する）。オペランドの具体値や呼び出し回数は縛らない。
- `discount_jacket` の `calculate` は引数指定なし（割引を `* 0.8` でも `- (×0.2)` でも到達でき、経路を縛りたくない）。

**合否基準**
- `response_numeric` の `tolerance` は丸め差を吸収する最小幅（`139.95` は `0.10`、`71.99` も `0.10`）。
- 1 タスクは全 check が 1.0 のときのみ pass。`price_shoes_synonym` のように「素では落ちるが改善後に通る」タスクをあえて入れ、**改善のビフォー/アフターを数値で見える化**するのが狙い。

### (c) `boutique` がどこで・なぜ壊れるか（このパートで露見する失敗）

素のエージェント（`SYSTEM_PROMPT="You are a helpful assistant."`、説明文が `"get_product"` / `"product"` だけの貧弱な tool specs、`return catalog[product]` の素の実装）では、各タスクは次のように壊れがちです。

- `price_tshirt`: tool spec にキー形式の説明が無いため、`tshirt` や `t shirt` を渡しうる。素の `get_product` は存在しないキーで **`KeyError`** を送出 → `execute_tool` が `"Error: 't shirt'"` を返し、`response_contains["24.99"]` と `arguments={"product":"t-shirt"}` の両方が落ちる可能性。
- `price_shoes_synonym`: `shoes` はカタログ外。素の実装では `KeyError`。spec に在庫一覧が無いので `sneakers` を代替提案できず `response_contains["sneakers"]` が落ちる。**設計上、素では FAIL する想定**。
- `total_shirts_belts`: system プロンプトに「暗算するな、必ず calculate を使え」が無いため、エージェントが**暗算**して `calculate` を呼ばない → `tool_use` の `*` / `+` チェックが落ちる。暗算ミスがあれば `response_numeric` も落ちる。
- `discount_jacket`: 同様に暗算しがち。さらに「20% off」を**値引後価格（71.99）**と**値引額（18.00）**のどちらに解釈するか揺れる → `response_numeric[71.99]` が落ちうる。

→ これらの失敗は**節 4 の改善**（system プロンプト・tool specs・エラーハンドリング）で解消する想定です。素の状態でおおむね半分前後の合格率（約 50%）、改善後にほぼ全通過（約 100%）へ、というのが原典の狙いです。

## 節 2. Part 6 — LLM-as-judge grader の実装（穴埋め: `grade_llm_judge`）

### (a) 完成コード

原典の `# TODO: Implement this grader` を、grader インターフェース（`(result, check, context=None) -> {"score", "reason"}`）に揃えて実装します。判定の安定のため、専用 system プロンプトで**1 行目に `PASS`/`FAIL`**を強制し、2 行目以降を理由として解析します。

```python
JUDGE_SYSTEM_PROMPT = """You are an eval grader. You will receive:
- The original user query
- An AI agent's response to that query
- A criterion to evaluate

Judge whether the agent's response meets the criterion. Focus only on the
specific criterion provided, not on overall response quality.

Respond with exactly one of these on the first line:
PASS - if the criterion is clearly met
FAIL - if the criterion is not met or only partially met

Then on the next line, give a brief reason (one sentence)."""


def grade_llm_judge(result, check, context=None):
    query = context["query"] if context else "Unknown query"
    response_text = result["final_text"]

    judge_prompt = f"""Original query: {query}

Agent's response: {response_text}

Criterion: {check}"""

    try:
        judge_response = client.messages.create(
            model="claude-haiku-4-5-20251001",
            max_tokens=150,
            temperature=0.0,
            system=JUDGE_SYSTEM_PROMPT,
            messages=[{"role": "user", "content": judge_prompt}],
        )
        judge_text = judge_response.content[0].text.strip()
        first_line = judge_text.split("\n")[0].strip().upper()
        reason = judge_text.split("\n", 1)[1].strip() if "\n" in judge_text else judge_text

        if "PASS" in first_line:
            return {"score": 1.0, "reason": f"LLM judge: {reason}"}
        elif "FAIL" in first_line:
            return {"score": 0.0, "reason": f"LLM judge: {reason}"}
        else:
            return {"score": 0.0, "reason": f"LLM judge returned unparseable response: {judge_text[:200]}"}

    except Exception as e:
        return {"score": 0.0, "reason": f"LLM judge error: {e}"}


# Register it so the runner can use it
GRADER_REGISTRY["llm_judge"] = grade_llm_judge
print(f"Graders now available: {list(GRADER_REGISTRY.keys())}")
```

> **モデル名の補足**: 原典の TODO コメントには `claude-haiku-4-5-20241022` という別表記が混在しますが、ファシリテーター参照実装および本ノートブックの `MODEL` 定義は `claude-haiku-4-5-20251001` です。判定モデルは本体と同じ `claude-haiku-4-5-20251001` に統一しています。

### (b) 設計意図

- **インターフェースの一致**: 決定的 grader と同じ `(result, check, context=None)` シグネチャ・同じ戻り値 `{"score": 0.0/1.0, "reason": ...}`。これにより `GRADER_REGISTRY["llm_judge"] = grade_llm_judge` の 1 行で runner にプラグインでき、`run_single_task` のループ（`grader_fn = GRADER_REGISTRY.get(grader["type"])`）がそのまま呼べる。
- **出力フォーマットの強制**: 自由文の判定は解析が壊れやすい。「1 行目に `PASS`/`FAIL`、2 行目に理由」と決め打ちし、`first_line` に `PASS`/`FAIL` が含まれるかで分岐。どちらでもなければ 0.0 + 「unparseable」理由を返し、**判定不能を黙って成功にしない**。
- **`temperature=0.0`**: 判定の再現性を高める（採点者はばらつくべきでない）。
- **atomic な criterion**: 1 回の judge 呼び出しで 1 つの基準だけを見る（複数基準を 1 プロンプトに詰めない）。後述のタスクで基準を 2 つに分けるのはこのため。各 `check` がそれぞれ独立した judge 呼び出しになる（runner が `checks` を 1 件ずつ grader に渡す）。
- **例外の握り込み**: API 失敗は `except` で捕捉し 0.0 を返す。ただしこれは「採点エラー」であって本来の `error`（インフラ障害）とは別レイヤー。LLM judge の API 失敗が混入するとノイズになりうる点は留意（本ノートでも触れる）。

### (c) 診断・改善との関係

LLM-as-judge は**改善前後どちらでも**機能しますが、判定が安定するのは「答えが妥当な複数解を持つ」場合です。`boutique` で壊れやすいのは判定側ではなく**本体側の出力**: 素の system プロンプトだと `What do you sell?` に対して「自分はただのアシスタント」的な**そっけない／文脈外**の応答を返し、judge の 2 番目の基準（shopping context で helpful か）が FAIL する可能性があります。これは節 4 の system プロンプト改善で解消します。

## 節 3. Part 6 — LLM-as-judge を使うタスクの追加（穴埋め: `llm_judge_tasks`）

### (a) 完成コード

原典のコメントアウトされた雛形を、`What do you sell?` 用に具体化します。`check` を 2 つに分け、それぞれ独立 judge 呼び出しにします（atomic）。

```python
llm_judge_tasks = [
    {
        "id": "what_do_you_sell",
        "description": "Open-ended: agent describes available products",
        "query": "What do you sell?",
        "category": "capabilities",
        "graders": [
            {"type": "llm_judge", "checks": [
                "Response describes or lists some of the available products in the catalog",
                "Response is helpful and relevant to a shopping context (not dismissive or off-topic)",
            ]},
        ],
    },
]

# 両方のタスクセットで eval を実行
all_tasks = tasks + llm_judge_tasks
results = run_eval(run_agent, all_tasks)
print_summary(results)
```

> 原典の参照実装では `what_do_you_sell` を `reference_tasks`（6 件）の一部として定義しています。受講者の `tasks`（5 件）に統合する場合は上記のように `tasks + llm_judge_tasks` で実行すれば、合計 6 タスクになります。

### (b) 設計意図

- **なぜ `What do you sell?` を LLM-as-judge にするか**: カタログの説明には正解文字列が一意に存在しない。「jeans, shirt, … を売っています」「衣料品全般を扱います」など多数の妥当解があるため、`response_contains` では取りこぼす／過検出する。意味的な合致は LLM judge が適任。
- **基準を 2 つに割る理由**: ①「カタログの商品を説明/列挙しているか」（内容の妥当性）と、②「shopping 文脈で helpful か、そっけなくないか」（トーン/関連性）は別軸。1 つの judge 呼び出しに混ぜると、片方だけ満たすケースの採点が曖昧になる。分割すれば各軸が独立に PASS/FAIL し、`print_summary` の reason でどちらが落ちたか分かる。
- **合否基準**: タスク pass = 2 つの judge check が両方 1.0。

### (c) 診断・改善との関係

素の `"You are a helpful assistant."` では、エージェントは自分が「ショッピングアシスタント」だと知らず、カタログにも触れないため基準①②が落ちやすい（特に②）。節 4 で `"You are Boutique, a shopping assistant…"` に変えると、役割と能力を自覚した応答になり両基準が通りやすくなります。

## 節 4. Part 5 — エージェントの改善（穴埋め: 失敗からの的を絞った修正）

eval の失敗（節 1〜3 の (c)）を根拠に、**3 点**を改善します。原典ファシリテーター参照と整合する完成形は次のとおり。

### (a) 完成コード

```python
# ── 修正 1: より良い system プロンプト ─────────────────────────────────────
SYSTEM_PROMPT = (
    "You are Boutique, a shopping assistant. You help customers find products, "
    "check prices, and calculate totals. Always use your tools to look up prices "
    "rather than guessing. If a product isn't found, suggest similar items from "
    "the catalog. Never do mental math, always use your calculate tool for any calculations."
)

# ── 修正 2: より良い tool specs ────────────────────────────────────────────
# 元の spec は "get_product" / "product" だけで、どの商品があるか・どんな
# 形式で渡すか・エラー時に何が起きるかを Claude に伝えていない。
GET_PRODUCT_SPEC = {
    "name": "get_product",
    "description": "Look up the price of a product from the store catalog. Returns the price as a number. Raises a KeyError if the product is not found.",
    "input_schema": {
        "type": "object",
        "properties": {
            "product": {
                "type": "string",
                "description": "Product name, lowercase. Available products: jeans, shirt, dress, jacket, sneakers, hat, socks, hoodie, shorts, t-shirt, sweater, belt",
            },
        },
        "required": ["product"],
    },
}

CALCULATE_SPEC = {
    "name": "calculate",
    "description": "Perform a math operation on two numbers. Use this for any arithmetic instead of doing mental math.",
    "input_schema": {
        "type": "object",
        "properties": {
            "op": {
                "type": "string",
                "description": "The math operator to apply.",
                "enum": ["+", "-", "*", "/", "**"],
            },
            "input1": {"type": "number", "description": "The first operand."},
            "input2": {"type": "number", "description": "The second operand."},
        },
        "required": ["op", "input1", "input2"],
    },
}

ALL_TOOL_SPECS = [GET_PRODUCT_SPEC, CALCULATE_SPEC]

# ── 修正 3: ツール実装のエラーハンドリング ─────────────────────────────────
# 元の get_product は catalog[product] で生の KeyError を投げる。
# 見つからないときは在庫一覧つきの分かりやすいメッセージを返す。
def get_product(product: str):
    catalog = {
        "jeans": 49.99, "shirt": 29.99, "dress": 59.99, "jacket": 89.99,
        "sneakers": 74.99, "hat": 19.99, "socks": 9.99, "hoodie": 44.99,
        "shorts": 34.99, "t-shirt": 24.99, "sweater": 54.99, "belt": 24.99,
    }
    if product in catalog:
        return catalog[product]
    available = ", ".join(sorted(catalog.keys()))
    return f"Product '{product}' not found. Available products: {available}"

TOOL_REGISTRY["get_product"] = get_product

print("Improved agent loaded. Re-run the eval to see the difference.")
```

> 改善後は **エージェントのセル（`run_agent` 等を定義したセル）を再実行**してから eval を回す点に注意（`SYSTEM_PROMPT` / `ALL_TOOL_SPECS` / `TOOL_REGISTRY` を `run_agent` が参照するため、上書き後に再評価する）。

### (b) 設計意図 — どの修正がどの失敗に効くか

| 修正 | 効く失敗 | 仕組み |
|------|----------|--------|
| **修正 1（system プロンプト）** | `total_shirts_belts` / `discount_jacket` の暗算、`what_do_you_sell` のトーン、`price_shoes_synonym` の代替提案 | 「役割（Boutique）」「価格は必ずツールで」「見つからなければ類似品を提案」「暗算禁止・計算は必ず calculate」を明示し、行動を方向づける |
| **修正 2（tool specs）** | `price_tshirt`（キー形式）、`price_shoes_synonym`（在庫の認識）、`calculate` の演算子ミス | spec に**在庫一覧と小文字指定**を書き、`t-shirt` を正しく渡せる。`op` に `enum`（`+ - * / **`）を付け不正演算子を防ぐ。説明文でエラー時挙動も共有 |

**決定的 grader と LLM judge の使い分け（全体方針の再掲）**: 一意解（価格・合計・割引）は決定的 grader、自由応答（capabilities）は LLM judge。改善は「grader を緩める」のではなく「エージェントを直す」方向で行うのが eval 駆動の要諦。

### (c) 期待される結果（改善前後の見立て）

- 改善前: `product_lookup` のハイフン/同義語、`multi_tool`・`calculation` の暗算、`capabilities` のトーンで落ち、おおむね半数前後（約 50%）。
- 改善後: 5〜6 タスクがほぼ全通過（約 100%）に近づく。`response_numeric` の許容差と `tool_use` の部分一致設計のおかげで、丸めや経路の細かな揺れでは落ちない。
- 非決定性のため `run_eval(run_agent, tasks, num_runs=5)` でベースラインを取り、改善後も複数回実行して**回帰がないか**を確認するのが正しい運用（原典 Part 4 の「baseline」セルの趣旨）。
