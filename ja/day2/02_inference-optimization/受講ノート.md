# 受講ノート — Inference Optimization

Claude モデルの推論性能を 4 指標（TTFT / TTC / OTPS / コスト）で計測し、プロンプトキャッシュとツール使用がそれらにどう効くかを、本番デプロイの最適化観点で学ぶハンズオン。

---

## ねらい

- 推論の性能を「体感速度・総待ち時間・スループット・コスト」の 4 つに分解して**定量計測**できるようになる。
- Haiku / Sonnet / Opus を同一プロンプトで横並び比較し、**適材適所のモデル選定**の判断軸を持つ。
- **プロンプトキャッシュ**でレイテンシとコストを下げる仕組みと条件を理解する。
- **ツール使用**が増やすラウンドトリップ・オーバーヘッドを実測の観点で把握する。

---

## キーコンセプト

| 概念 | ひとことで |
|---|---|
| TTFT（最初のトークンまでの時間） | ユーザーが体感する「反応の速さ」 |
| TTC（完了までの時間） | リクエスト全体の総所要時間 |
| OTPS（1秒あたり出力トークン数） | ストリーミング開始**後**の生成スループット |
| コスト | 入力/出力トークンの単価別課金（出力が割高） |
| ストリーミング計測 | `content_block_start` で TTFT、ストリーム終了で TTC を計時 |
| プロンプトキャッシュ | 不変の接頭辞を再利用し再処理をスキップ（5 分・1024 トークン以上） |
| ツール使用オーバーヘッド | API 往復が 2 回に増え、履歴再送でトークンも増える |

---

## 指標チートシート

| 指標 | 定義 / 計算式 | 何を表すか | 主な最適化レバー |
|---|---|---|---|
| **TTFT** | リクエスト送出 → 最初のトークン受信までの秒数。`content_block_start` 受信時に確定 | 体感速度。チャット UI で最重要 | 小さいモデル（Haiku）、ストリーミング、プロンプトキャッシュ、入力短縮 |
| **TTC** | リクエスト送出 → ストリーム完了までの秒数（`total_time`） | 総待ち時間。バッチ処理で重要 | 出力短縮（`max_tokens`）、軽量モデル、キャッシュ |
| **OTPS** | `output_tokens / (TTC − TTFT)` | 生成の吐き出し速度。TTFT を引くのは待ち時間を除くため | モデル選択、（短応答だと不安定なので）長め生成で計測 |
| **コスト** | `input_tokens/1e6 × 入力単価 + output_tokens/1e6 × 出力単価` | 1 呼び出しのドルコスト。`summary` は `$/1K calls` 表示 | 出力削減、軽量モデル、プロンプトキャッシュ、ツール定義の最小化 |

プライシング（1M トークンあたり USD・原文保持）:

| モデル | input | output |
|---|---|---|
| `claude-sonnet-4-6` | 3.00 | 15.00 |
| `claude-haiku-4-5-20251001` | 1.00 | 5.00 |
| `claude-opus-4-7` | 5.00 | 25.00 |

いずれも **出力は入力の 5 倍単価**。コストは出力長に支配されやすい。

## プロンプトキャッシュの要点

- 付け方: content ブロックに `"cache_control": {"type": "ephemeral"}`。
- 課金: 初回書き込みは +25%、以降の読み込みは −90%。
- 条件: 最低 **1,024 トークン**、寿命 **5 分**、**接頭辞の完全一致**。
- 配置: **不変の大きな塊を先頭**（システムプロンプト/ツール定義/ドキュメント/Few-shot）、可変部（質問）は後ろ。
- マルチターン: 直前のアシスタント発話に `cache_control` を移動 = 「ここまでをキャッシュ」。ターンが進むほど `cache_read_input_tokens`（`cached`）が増える。

---

## よくある落とし穴 / ベストプラクティス

- **OTPS で TTFT を引き忘れる**: 待ち時間を含めると生成速度を過小評価する。式は `output / (TTC − TTFT)`。
- **コスト単価の桁ミス**: 単価は 1M トークン基準。`/ 1_000_000` を忘れない。
- **短い応答で OTPS を語る**: 分母（生成時間）が小さくノイズが拡大。安定値が欲しければ長め生成 + 反復測定。
- **キャッシュが効かない**: 1,024 トークン未満、接頭辞が 1 文字でも変化、5 分超過 のいずれか。可変部を先頭に置くと毎回キャッシュミス。
- **キャッシュブレークポイントの付けっぱなし**: マルチターンでは前ターンの `cache_control` を素テキストに戻してから最新発話に付け直す（境界を会話末尾に追従させる）。
- **不要なツールを毎回同梱**: ツール定義は入力トークンを食う。使うツールだけ渡す。
- **計測の起点ずれ**: 計時は `time.perf_counter()` を API 呼び出し直前に置く。
- **1 回計測で判断**: TTFT/TTC は外れ値が出やすい。平均/中央値、SLA は p95（`percentile()`）で見る。
- **ヘルスチェックは `max_tokens` 小**: 接続確認で課金と待ちを最小化。失敗は `anthropic.APIError` で早期検知。

## 持ち帰りポイント

1. 推論性能は **TTFT（体感）・TTC（総時間）・OTPS（吐き出し速度）・コスト** に分解して測る。OTPS は必ず TTFT を引いた生成時間で割る。
2. **モデルは適材適所**。Haiku=速く安い軽量タスク、Sonnet=汎用バランス、Opus=最難関のみ。出力単価が高いので出力長の制御がコストに直結。
3. **プロンプトキャッシュを既定で使う**。不変の大きな前置きを先頭に置き `cache_control` を付ける（1,024 トークン以上・5 分・接頭辞完全一致）。マルチターンはブレークポイントを最新発話へ移動。
4. **ツール使用は往復とトークンを増やす**。正確性・外部作用が要る箇所だけに使い、不要なツール定義は渡さない。
5. **計測はノイズ込みで評価**。反復して平均/中央値、SLA は p95 を見る。短応答の OTPS は信用しすぎない。
