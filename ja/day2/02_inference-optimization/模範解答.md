# 模範解答 — Inference Optimization（推論最適化）

本ドキュメントは `day2/02_inference-optimization/Inference_Optimization.ipynb`（および同等の `.py`）の演習に対する模範解答と解説です。Claude モデルの推論性能を 4 つの指標 — 最初のトークンまでの時間（TTFT: Time to First Token）、完了までの時間（TTC: Time to Completion）、1 秒あたりの出力トークン数（OTPS: Output Tokens Per Second）、コスト（Cost） — で計測し、さらにプロンプトキャッシュ（prompt caching）とツール使用（tool use）がそれらに与える影響を測ります。

> 注記: 本リポジトリでは API キーが未設定（`api_key = ""`）のため、ベンチマークは**未実行**です。以下のコードと数値の解釈は、原典コードの仕様とプライシング表（`PRICING`）から導いた説明であり、実測値ではありません。実際の TTFT/TTC/OTPS は時間帯・リージョン・負荷で変動します。

計測結果は `BenchmarkResult` データクラスに格納し、`BenchmarkSuite` で集計します（このクラスは改変不要・理解のみ）。

```python
@dataclass
class BenchmarkResult:
    """Timing, tokens, and cost for a single API call."""
    ttft: float                    # Time to First Token (seconds)
    total_time: float              # Time to Completion / TTC (seconds)
    tokens_per_second: float       # Legacy: output_tokens / ttc
    input_tokens: int
    output_tokens: int
    endpoint: str
    model: str
    test_name: str
    otps: Optional[float] = None   # Output Tokens Per Second
    cost: Optional[float] = None   # Cost in dollars
    cache_creation_input_tokens: Optional[int] = None
    cache_read_input_tokens: Optional[int] = None
```

`BenchmarkSuite.summary()` は `test_name` でグループ化し、TTFT(ms)・TTC(ms)・OTPS の平均と、`$/1K calls`（1000 回呼び出しあたりのコスト = `sum(costs)*1000`）を `tabulate` のグリッド形式で表示します。1 回あたりのコストは極小（小数第 6 位）なので、1000 回換算にすることで読みやすくしているのがポイントです。

## 2. ストリーミング計測ヘルパー `_stream_request`（Part 1）

すべての計測は同じストリーミングパターン「ストリームを開く → イベントを監視 → タイミングを記録」に基づきます。これを再利用可能なヘルパーに切り出したものが `_stream_request` です（このセルは実装済み、模範として読み取り方を示します）。

```python
def _stream_request(prompt, model=DEFAULT_MODEL, max_tokens=256):
    """Low-level helper: stream a request, return raw timing + response."""
    ttft = None
    start_time = time.perf_counter()
    with client.messages.stream(
        model=model, max_tokens=max_tokens,
        messages=[{"role": "user", "content": prompt}],
    ) as stream:
        for event in stream:
            if ttft is None and event.type == "content_block_start":
                ttft = time.perf_counter() - start_time
        response = stream.get_final_message()
    total_time = time.perf_counter() - start_time
    return ttft, total_time, response
```

読み取りのポイント:

- **計時の起点** は `client.messages.stream(...)` を呼ぶ直前の `time.perf_counter()`。`perf_counter` は単調増加・高分解能で、経過時間の計測に適切（`time.time()` よりも望ましい）。
- **TTFT** は最初の `content_block_start` イベントを受け取った瞬間に確定させる。`ttft is None` ガードで「最初の 1 回だけ」記録するのが重要。これがリクエスト送出から最初のトークンが返り始めるまでの待ち時間で、ユーザーが体感する「反応の速さ」に直結する。
- **TTC（total_time）** はストリーム終了後（`get_final_message()` 取得後）の経過時間。リクエスト全体の所要時間。
- `stream.get_final_message()` で `usage`（`input_tokens` / `output_tokens` / キャッシュ系トークン）を含む完成レスポンスを取得できる。

## 3. 穴埋め① OTPS の計算 `compute_otps`（Part 1）

OTPS は「ストリーミング開始後」の生成スループットを表します。計算式は次の通り（マークダウンの Formulas より）。

> **OTPS** = `output_tokens / (ttc - ttft)`

TTFT を引く理由は、最初のトークンが来るまでの時間は「待ち時間」であって「生成時間」ではないためです。生成にかかった実時間（generation time）だけで割ることで、純粋な吐き出し速度を測れます。

### 完成コード

```python
def compute_otps(ttft, total_time, output_tokens):
    """Return (otps, generation_time).

    OTPS = output_tokens / (ttc - ttft)
    生成時間 = TTC - TTFT（最初のトークンが来るまでの待ち時間を除外）
    """
    gen_time = total_time - ttft
    if gen_time <= 0 or output_tokens <= 0:
        # ゼロ除算/負値の防御。極小レスポンスや計測ノイズへの保険。
        return 0.0, gen_time
    otps = output_tokens / gen_time
    return otps, gen_time
```

呼び出し側の穴埋めも合わせて完成させます。

```python
ttft, total_time, response = _stream_request("What is 2 + 2? Answer in one word.")

tokens = response.usage.output_tokens          # 出力トークン数を usage から取得
otps, gen_time = compute_otps(ttft, total_time, tokens)

print(f"Response: {response.content[0].text}")
print(f"TTFT: {ttft * 1000:.0f}ms")
print(f"TTC:  {total_time * 1000:.0f}ms")
print(f"OTPS: {otps:.1f} tokens/sec ({tokens} tokens / {gen_time:.3f}s)")
```

注意: 短い応答（数トークン）では OTPS のばらつきが大きくなります。分母 `gen_time` が小さいため、計測ノイズが相対的に拡大するからです。安定した OTPS が欲しいときは長めの生成で測ります。

## 4. 穴埋め② コスト計算 `calculate_cost`（Part 1）

入力トークンと出力トークンは別単価で課金されます。プライシングは 100 万トークンあたりのドル単価で定義されています（キー・数値・モデル名は原文保持）。

```python
PRICING = {
    "claude-sonnet-4-6": {"input": 3.00, "output": 15.00},
    "claude-haiku-4-5-20251001": {"input": 1.00, "output": 5.00},
    "claude-opus-4-7": {"input": 5.00, "output": 25.00},
}
```

### 完成コード

```python
def calculate_cost(model: str, input_tokens: int, output_tokens: int) -> tuple[float, float, float]:
    """Return (input_cost, output_cost, total_cost) in USD.

    PRICING は 1M（100万）トークンあたりの USD 単価なので 1_000_000 で割る。
    """
    prices = PRICING.get(model, {"input": 3.00, "output": 15.00})
    input_cost = input_tokens / 1_000_000 * prices["input"]
    output_cost = output_tokens / 1_000_000 * prices["output"]
    total = input_cost + output_cost
    return input_cost, output_cost, total
```

読み取りのポイント:

- 単価は **1M トークン基準**。トークン数を `1_000_000` で割ってから単価を掛ける。
- 出力トークンは入力の 5 倍（各モデルとも `output = input × 5`）。**出力の長さがコストを支配しやすい**ため、`max_tokens` の抑制や簡潔な指示が直接効く。
- 未知のモデル名に対しては Sonnet 相当（`{"input": 3.00, "output": 15.00}`）にフォールバックする設計。

## 5. 計測の合成 `measure_streaming_latency`（Part 1）

`_stream_request` / `compute_otps` / `calculate_cost` を 1 つにまとめる関数です（実装済み、新規ロジックなし）。完成した `compute_otps` と `calculate_cost` がここで初めてすべて噛み合います。

```python
def measure_streaming_latency(prompt, model=DEFAULT_MODEL, max_tokens=256, test_name="streaming"):
    ttft, total_time, response = _stream_request(prompt, model, max_tokens)
    usage = response.usage
    _, _, cost = calculate_cost(model, usage.input_tokens, usage.output_tokens)
    otps, _ = compute_otps(ttft, total_time, usage.output_tokens)
    return BenchmarkResult(
        ttft=ttft, total_time=total_time, tokens_per_second=otps,
        input_tokens=usage.input_tokens, output_tokens=usage.output_tokens,
        endpoint="1p", model=model, test_name=test_name,
        otps=otps, cost=cost,
    )
```

`BenchmarkResult` の `tokens_per_second` フィールドにも `otps` を入れているため、`summary()` の `throughputs = [r.otps or r.tokens_per_second ...]` が常に正しく機能します。

## 6. モデル比較 Haiku / Sonnet / Opus（Part 2）

同一プロンプト `"What is machine learning? Answer in 2 sentences."` を 3 モデル × 5 回ずつ流して性能プロファイルを比較します（実装済み）。

```python
models = [
    (MODEL_HAIKU, "haiku"),
    (MODEL_SONNET, "sonnet"),
    (MODEL_OPUS, "opus"),
]
for model_id, model_name in models:
    for i in range(5):
        result = measure_streaming_latency(PROMPT, model=model_id, test_name=model_name)
        suite.add_result(result)
print(suite.summary())
```

### 解釈（最適化指針）

- **5 回反復する理由**: TTFT/TTC は外れ値が出やすいため、平均（必要なら中央値や `percentile()` での p50/p95）で評価する。ヘルパー `percentile()` が用意されているのは、本番の SLA はしばしば p95 で語られるため。
- **一般的な傾向（モデルクラスの設計目標）**:
  - **Haiku** — 最速・最安。単価 `input 1.00 / output 5.00`。分類・抽出・ルーティング・大量バッチなど、レイテンシとコストが効く軽量タスク向け。
  - **Sonnet** — 速度・知能・コストのバランス（既定モデル）。単価 `input 3.00 / output 15.00`。汎用ワークロードの第一候補。
  - **Opus** — 最高難度の推論向けだが最も高価。単価 `input 5.00 / output 25.00`。難しい一部のステップにだけ使い、残りは下位モデルに振り分ける（モデルカスケード）と費用対効果が高い。
- **読み取り方**: TTFT が体感速度、TTC が総待ち時間、OTPS が吐き出し速度。`$/1K calls` で同一タスクの相対コストを横並び比較する。「最速」「最安」「最賢」は両立しないため、タスクごとに**どの指標を最適化するか**を先に決めるのが定石。

## 7. ツール使用のオーバーヘッド計測（Part 3）

ツール使用はラウンドトリップが 1 往復増えます: **Request → tool_use → Execute → Result → Response**。まず JSON Schema でツールを定義します。

### 穴埋め③ ツールスキーマ `CALCULATOR_TOOL`

```python
CALCULATOR_TOOL = {
    "name": "calculator",
    "description": "Performs basic arithmetic operations.",
    "input_schema": {
        "type": "object",
        "properties": {
            "operation": {
                "type": "string",
                "enum": ["add", "subtract", "multiply", "divide"],
                "description": "The arithmetic operation to perform.",
            },
            "operands": {
                "type": "array",
                "items": {"type": "number"},
                "description": "Exactly two numbers to operate on, e.g. [42, 17].",
            },
        },
        "required": ["operation", "operands"],
    },
}
```

ポイント:

- `enum` でローカル実装 `execute_calculator`（`add`/`subtract`/`multiply`/`divide`）と一致させる。モデルが不正な operation を選ぶ余地を減らす。
- `operands` は数値配列。`description` を明確にするとモデルの引数生成精度が上がる。

### 穴埋め④ ラウンドトリップ計測 `measure_tool_use_latency`

往復処理の全体（1. ツール付き送信 → 2. tool_use 解析 → 3. ローカル実行 → 4. 結果返送 → 5. 最終応答）を実装します。

```python
def measure_tool_use_latency(prompt, model=DEFAULT_MODEL, max_tokens=256):
    """Measure full round-trip latency for a tool use request."""
    start_time = time.perf_counter()

    # 1) ツール付きの最初の API 呼び出し
    first_response = client.messages.create(
        model=model,
        max_tokens=max_tokens,
        tools=[CALCULATOR_TOOL],
        messages=[{"role": "user", "content": prompt}],
    )
    ttft = time.perf_counter() - start_time

    # 2) content の中から tool_use ブロックを探す
    tool_use_block = next(
        (b for b in first_response.content if getattr(b, "type", None) == "tool_use"),
        None,
    )

    if tool_use_block is None:
        # ツールが使われなかった場合は早期 return
        total_time = time.perf_counter() - start_time
        return (ttft, total_time, "No tool used",
                first_response.usage.input_tokens, first_response.usage.output_tokens)

    # 3) ツールをローカル実行
    result = execute_calculator(
        tool_use_block.input["operation"],
        tool_use_block.input["operands"],
    )

    # 4) tool_result を含めて 2 回目の呼び出し
    second_response = client.messages.create(
        model=model,
        max_tokens=max_tokens,
        tools=[CALCULATOR_TOOL],
        messages=[
            {"role": "user", "content": prompt},
            {"role": "assistant", "content": first_response.content},
            {"role": "user", "content": [
                {
                    "type": "tool_result",
                    "tool_use_id": tool_use_block.id,
                    "content": str(result),
                }
            ]},
        ],
    )

    total_time = time.perf_counter() - start_time

    final_text = ""
    for block in second_response.content:
        if hasattr(block, "text"):
            final_text += block.text

    total_input = first_response.usage.input_tokens + second_response.usage.input_tokens
    total_output = first_response.usage.output_tokens + second_response.usage.output_tokens
    return ttft, total_time, final_text, total_input, total_output
```

実装上の重要ポイント:

- 最初の呼び出しは**非ストリーミング**（`create`）。ここでの `ttft` は厳密な「最初のトークン」ではなく「最初のレスポンスが返るまで」の意味で使われている点に注意。ツール使用の往復全体の構造把握が目的。
- **会話履歴の積み上げが必須**: 2 回目の呼び出しでは、`user`（元の質問）→ `assistant`（`first_response.content` 全体、tool_use ブロックを含む）→ `user`（`tool_result`）を順に渡す。`tool_use_id` で結果と呼び出しを対応付ける。
- **トークンが二重計上される**: 2 回目は会話履歴全部を再送するため `input_tokens` が膨らむ。これがツール使用のコスト面のオーバーヘッドの正体。

### ツールあり vs なしの比較（Part 3 末尾）

同じ計算課題を「ツールなし（Sonnet に筆算させる）」と「ツールあり（calculator を使わせる）」で 5 回ずつ計測し、`suite.summary()` で並べます。

解釈:

- **TTFT/TTC はツールありの方が大きくなる**: API 往復が 2 回に増えるため、ネットワーク往復とモデル処理が累積する。レイテンシ面では純粋にオーバーヘッド。
- **入力トークン（コスト）も増える**: 履歴とツール定義を再送するため。
- **では何のためのツールか**: 正確性・決定性・外部作用（DB 参照、API 実行、計算の厳密性）のため。レイテンシ/コストと引き換えに「信頼できる結果」を得る。**速度最優先なら可能な範囲でツールを避け、正確性が要るところだけツールを使う**のが指針。ツール定義を増やすほど入力トークンが増えるので、不要なツールはリクエストに含めない。

## 8. プロンプトキャッシュ（Part 4）

処理済みプロンプトをキャッシュし、後続呼び出しで再読込をスキップさせます。

仕組み（マークダウンより）:

1. content ブロックに `cache_control: {"type": "ephemeral"}` を付ける
2. 初回はキャッシュへ**書き込み**（キャッシュ対象トークンは 25% 割増）
3. 以降はキャッシュから**読み込み**（キャッシュ対象トークンは 90% 割安）

要件:

- キャッシュブロックは最低 **1,024 トークン**必要
- ephemeral キャッシュの寿命は **5 分**
- キャッシュ内容は**完全な接頭辞一致**が必要（プレフィックスが 1 文字でも変われば失効）

### 穴埋め⑤ システムプロンプトのキャッシュ `cached_request`

`SYSTEM_PROMPT` は `... * 20` で 1,024 トークン超を確保しています（要件を満たすため）。

```python
# cache_control を有効にした system_block を作る
system_block = [
    {
        "type": "text",
        "text": SYSTEM_PROMPT,
        "cache_control": {"type": "ephemeral"},
    }
]

def cached_request(question):
    start = time.perf_counter()
    response = client.messages.create(
        model=DEFAULT_MODEL,
        max_tokens=256,
        system=system_block,
        messages=[{"role": "user", "content": question}],
    )
    elapsed = time.perf_counter() - start
    return response, elapsed
```

読み取り方（コールド vs ウォーム）:

- **Call 1（Cold, "What is REST?"）**: `cache_creation_input_tokens` が大きく（システムプロンプト分を書き込み）、`cache_read_input_tokens` は 0。初回は割増コスト。
- **Call 2（Warm, "What is OAuth?"）**: 質問は違うが**システムプロンプトは同一接頭辞**なので、`cache_read_input_tokens` が大きく、`cache_creation_input_tokens` は 0。`input_tokens`（新規課金対象）はユーザーの質問分だけに激減し、TTFT/TTC も短縮される。
- キャッシュが効く条件は「変わらない大きな塊を**先頭**に置く」こと。質問のような可変部分は後ろに置く。

### 穴埋め⑥ マルチターンのキャッシュブレークポイント `chat`

会話が伸びるたびに過去メッセージを毎ターン再処理してしまう問題を、キャッシュブレークポイントで解消します。**直前のアシスタント発話に `cache_control` を付ける** = 「ここまでをキャッシュせよ」という指示です。システムプロンプトは `SYSTEM`（`cache_control` 付き）で常にキャッシュ済み。

```python
def chat(messages, new_question):
    # 古いキャッシュブレークポイントを掃除（list content を素のテキストに戻す）
    for msg in messages:
        if msg["role"] == "assistant" and isinstance(msg["content"], list):
            msg["content"] = msg["content"][0]["text"]

    # 直前の assistant メッセージにキャッシュブレークポイントを設定
    if messages and messages[-1]["role"] == "assistant":
        messages[-1]["content"] = [
            {
                "type": "text",
                "text": messages[-1]["content"],
                "cache_control": {"type": "ephemeral"},
            }
        ]

    messages.append({"role": "user", "content": new_question})

    start = time.perf_counter()
    response = client.messages.create(
        model=MODEL_SONNET,
        max_tokens=300,
        system=SYSTEM,
        messages=messages,
    )
    elapsed = time.perf_counter() - start

    answer = response.content[0].text
    messages.append({"role": "assistant", "content": answer})
    return answer, elapsed, response.usage
```

読み取り方:

- **なぜ毎ターン「掃除 → 付け直し」するのか**: ブレークポイントは「最新の確定済み履歴の末尾」に動かしたい。前ターンで付けた `cache_control` を素のテキストに戻し、最新のアシスタント発話に付け替えることで、キャッシュ境界を会話の伸長に追従させる。
- **観察ポイント**: ターンが進むほど `cached`（= `cache_read_input_tokens`）が増えていく。直前ターンまでのシステム + 会話履歴がキャッシュヒットし、毎ターン新規処理されるのは新しい質問とその応答だけになる。
- これにより、長い会話でもターンあたりのレイテンシとコストがほぼ一定に保たれる（履歴に比例して悪化しない）。

---

## 9. 本番デプロイの最適化指針（まとめ）

- **指標の優先順位を先に決める**: チャット UI は TTFT 最優先、バッチ処理は TTC/コスト最優先、長文生成は OTPS 重視。タスクごとに最適化対象を変える。
- **モデルは適材適所**: 軽量タスクは Haiku、汎用は Sonnet、最難関ステップだけ Opus。出力トークン単価が高いので、`max_tokens` と簡潔指示でコストを抑える。
- **プロンプトキャッシュを既定で使う**: 変わらない大きな前置き（システムプロンプト、ツール定義、ドキュメント、Few-shot 例）は先頭に置き `cache_control` を付ける。1,024 トークン未満では効かない点、5 分で失効する点、接頭辞完全一致が条件である点に注意。マルチターンでは直前のアシスタント発話にブレークポイントを移動させて履歴をキャッシュする。
- **ツール使用は往復コストを意識**: レイテンシとトークンが増える。正確性・外部作用が必要なところだけ使い、不要なツール定義はリクエストから外す。
- **計測はノイズ込みで評価**: 反復測定して平均/中央値、必要なら p95（`percentile()`）を見る。短い応答での OTPS は不安定。
